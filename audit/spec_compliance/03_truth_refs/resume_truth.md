# Resume Truth — Canonical Durability and Crash-Resume Facts

Flat fact list for the resume regime. Each line is one verifiable fact with a `<!-- source: ... -->` citation. Used by Phase 7 (resume crossref) to grep against `resume_claims.jsonl`.

- HF Hub commits via `HfApi.upload_folder` / `upload_large_folder` are durable the moment the HTTP commit returns 200 OK. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_jobs_durability.md -->
- `upload_large_folder` is resumable: each task (hash, preupload, commit) is checkpointed locally under `./cache/huggingface/`, so an interrupted upload restarts from the last completed task. <!-- source: https://huggingface.co/docs/huggingface_hub/guides/upload (fetched 2026-05-01) -->
- A single `create_commit` (the primitive behind `upload_file` / `upload_folder`) is the atomic commit unit on the Hub side; until that call returns, no partial state is visible to readers, and after it returns the new revision is the durable system-of-record. <!-- source: https://huggingface.co/docs/huggingface_hub/guides/upload (fetched 2026-05-01) -->
- HF Jobs bucket FUSE mounts use hf-mount semantics with two write modes — streaming (buffered, lost on crash before `close()`) and advanced-writes (staged + async flush, lost on crash within the 2–30 s flush window) — so bucket writes are NOT durable on cancel/timeout. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_jobs_durability.md -->
- `hf jobs cancel` cancellation grace period is undocumented; treat it as worst case (SIGKILL, no grace), which discards FUSE write-back caches. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_jobs_durability.md -->
- The bucket is scratch space for in-pod intermediates only; it must never be the system of record. The Hub repo is the only durable boundary. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/feedback_hf_jobs_bucket_cancel.md -->
- HF Jobs default timeout is 30 minutes when `--timeout` is omitted; the job ends with `status=ERROR, message="Job timeout"` and the pod is terminated identically to `hf jobs cancel`. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/feedback_hf_jobs_timeout.md -->
- "Job timeout" is equivalent to "cancelled mid-run" for durability planning — both kill the pod and discard non-Hub state. Always pass `--timeout` explicitly, sized to workload + headroom (e.g. 8 h per heavy stage). <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/feedback_hf_jobs_timeout.md -->
- Per-stage durability policy: each pipeline stage producing a durable artifact must call `upload_large_folder` to a Hub repo before the job exits. Multi-stage runs use `STOP_AFTER_STAGE=N` with one job per stage so each stage's output is committed before the next begins. On resume, download from the Hub (via `PRIOR_STAGE_REPO`), never the bucket. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_jobs_durability.md -->
- Spec §11 mandates per-stage Hub commit as the canonical durability boundary; partial checkpoints under `_stage{N}_partial/` are local-FS scratch only. <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 lines 678–722 -->
- Local-FS atomic write idiom: write payload to `*.tmp`, `fsync` the file (and parent directory), then `os.rename` / `os.replace` to the final name. A SIGKILL mid-write leaves at most a `.tmp`, never a truncated final file. <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 line 695 -->
- POSIX `rename(2)` atomicity: "If newpath already exists, it will be atomically replaced, so that there is no point at which another process attempting to access newpath will find it missing." <!-- source: https://man7.org/linux/man-pages/man2/rename.2.html (fetched 2026-05-01) -->
- POSIX `rename(2)` does NOT cross filesystems/mounts: "oldpath and newpath are not on the same mounted filesystem … rename() does not work across different mount points, even if the same filesystem is mounted on both." Cross-mount rename returns `EXDEV` and atomicity does not apply. <!-- source: https://man7.org/linux/man-pages/man2/rename.2.html (fetched 2026-05-01) -->
- `fsync` of the file (and parent directory) is required BEFORE rename for crash-durability; without fsync, the rename can be reordered with the data write and a power-loss / kernel-panic crash can leave the renamed file pointing at zero-length or stale data. <!-- source: https://man7.org/linux/man-pages/man2/rename.2.html (fetched 2026-05-01); cross-ref ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_jobs_durability.md -->
- NFS caveat: on NFS filesystems atomicity cannot be assumed even within the same mount — "you can not assume that if the operation failed, the file was not renamed." Resume logic should not rely on rename atomicity over NFS. <!-- source: https://man7.org/linux/man-pages/man2/rename.2.html (fetched 2026-05-01) -->
- File ordering invariant: write the payload (`.pt`) BEFORE the manifest (`.json`), each via `.tmp`+rename. Existence of the `.json` then implies the `.pt` payload is fully on disk. The reverse order admits a race where the manifest references an absent or partial payload. <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 line 695 (atomic checkpointing) + line 489 (Stage 4 manifest semantics) -->
- Idempotence requirement: the per-layer / per-stage completion check must be safe to re-run after a crash. On resume, completed layers are loaded directly from `_stage{N}_partial/`; failed layers re-run from the prior stage's output. Re-running a completed stage with no partial deltas is a no-op. <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 lines 678–697; line 489 (Stage 4 resume) -->
- Stage 6 idempotence: Stage 6 is stateless. Re-running is always safe; the teacher eval cache persists across runs as a JSON file in artifacts (also uploaded to Hub). <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §9 line 660 -->
- Dangling `.tmp` cleanup: stage startup must remove leftover `.tmp` files from a prior killed run before any new partial-write begins. <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 line 695 -->
- `format_version` field is mandatory on every partial checkpoint; on resume the version is checked before any state is restored, and a mismatch raises an actionable error ("delete `_stage{N}_partial/` and re-run"). This guards against silently re-loading stale checkpoints across spec/code changes. <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 line 722; §7 line 489 (`format_version=1`) -->
- `--no-resume` flag disables all within-stage resume behaviour: each stage runs unconditionally from scratch with no partial-file I/O. Stage 1 and Stage 6 are unaffected (they have no resume files). <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §11 line 697 -->
- Teacher eval cache key = `sha256(model_name + revision + eval_config_subset)`, providing automatic invalidation when any input changes. First run populates the cache; subsequent runs load directly and skip teacher load + evaluation (~50% Stage 6 wall-clock saved). <!-- source: /home/lucas/ai/moe_compress/max_quality/ALGORITHM_REFERENCE.md §9 lines 599–602 -->
- Calibration durability/freshness rule: when the token budget exceeds the cached calibration set, increase `calibration.num_sequences` to draw fresh disjoint samples — never wrap-around / `itertools.cycle` over already-seen tokens. Reuse trades quality for nothing because Cascade-2 has millions of disjoint rows per subset. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/feedback_calibration_data.md -->
- HF token usability: token at `~/.cache/huggingface/token` is only usable via the Python SDK in `~/ai/venv` (`huggingface_hub.get_token()`); git HTTPS auth fails with "expected acknowledgments". Use `snapshot_download` for resume-time downloads of prior-stage Hub repos. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_token.md -->
- `CommitScheduler` is for append-only logs / feedback files, NOT for multi-GB checkpoints — wrong tool for stage durability in this pipeline. <!-- source: ~/.claude/projects/-home-lucas-ai-moe-compress/memory/reference_hf_jobs_durability.md ; cross-ref https://huggingface.co/docs/huggingface_hub/guides/upload (fetched 2026-05-01) -->
