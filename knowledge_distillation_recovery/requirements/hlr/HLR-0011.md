## Description

The YAML SHALL contain exactly one `calibration` block (source dataset, sequence length, total sequences, subset weights). In `da_qad` mode the first N=256 to N=512 sequences (configurable, default 256) feed `mtq.quantize`'s `calibrate_forward_loop` to set initial fake-quant scales; the full token budget (`distillation.total_tokens`) feeds the FKLD distillation loop. Both loops read from the same packed calibration tensor produced by `moe_compress.utils.calibration._tokenize_to_fixed_length`. There SHALL NOT be a separate `ptq_calibration` block in the YAML.

## Rationale

Drift between calibration data and training data is a known source of subtle QAT instability — initial scales calibrated on one distribution and gradients flowing on another can produce weight values that are correct for neither. Sharing the source eliminates this category of bug at zero engineering cost. The first-N-sequences split rather than a separate dataset is justified by modelopt's calibrate_forward_loop only requiring a few hundred batches to set scales; using more would waste calibration time without improving scale quality.

## Acceptance Criteria

- [ ] `kdr.config.CalibrationConfig` exposes `source`, `dataset`, `seed`, `num_sequences`, `sequence_length`, `subset_weights` and an optional `ptq_subset_size: int = 256`
- [ ] In `da_qad` mode, modelopt's `calibrate_forward_loop` consumes exactly `ptq_subset_size` sequences from the front of the calibration tensor
- [ ] In both modes, FKLD distillation iterates the full calibration tensor up to `distillation.total_tokens`
- [ ] If a YAML attempts to define a separate `ptq_calibration` block, Pydantic raises a ValidationError naming the field as unknown
