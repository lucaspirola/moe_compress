## Description

In `da_qad` mode the YAML `quant` block SHALL carry a `weight` sub-block, a `kv_quant.key` sub-block, and a `kv_quant.value` sub-block. Each sub-block independently specifies `bits`, `format` (one of `int | fp8 | mxfp4 | nvfp4`), `granularity` (one of `tensor | channel | group | block | token`), and `transform`. In v0 `transform` accepts ONLY the literal `none` — Hadamard and FWHT transforms are deferred to v1+ per the plan's "Deferred" section. Defaults SHALL NOT be silently injected; every leaf field is explicit in the YAML. Asymmetric configurations (e.g., 4-bit K + 2-bit V; INT8-K + 3-bit V) SHALL be expressible without code changes. Adding new transforms in v1+ SHALL require only extending the `Transform` Literal enum and routing logic in NativeBackend; no other LLR needs to change.

## Rationale

Recent literature has moved past symmetric K/V configurations: KV-AdaQuant (arxiv:2502.15075) shows a 20-point accuracy swing between 4-bit-K-2-bit-V (75.2%) and the reverse (54.7%); PolyKV (arxiv:2604.24971) uses int8 K + 3-bit log-space V; TurboAngle (arxiv:2603.27467) uses 8-bit K + 4-bit log V with per-layer budgets. A symmetric `kv_bits` field would lock kdr out of every recipe in this lineage. The strict-no-defaults rule prevents recipes from drifting between configs that look the same but quantize differently, which is exactly the paper-spec-code coherence problem the req tool exists to prevent.

## Acceptance Criteria

- [ ] `kdr.quant.specs.WeightQuantSpec` and `KVQuantSpec` are Pydantic v2 strict models with all four fields required
- [ ] YAML loading raises a Pydantic ValidationError if any leaf field (`bits`, `format`, `granularity`, `transform`) is missing from `kv_quant.key` or `kv_quant.value`
- [ ] At least one ship-with config in `configs/` exercises asymmetric K/V (e.g., `zaya1_8b_da_qad_int4k_int2v.yaml`)
- [ ] Format enum literally includes `int`, `fp8`, `mxfp4`, `nvfp4`; granularity enum literally includes `tensor`, `channel`, `group`, `block`, `token`; transform enum in v0 contains exactly `none` (no Hadamard / FWHT)
- [ ] A property test on the spec model exercises the cross-product of (formats × granularities) per K/V independently and shows no validation crashes
