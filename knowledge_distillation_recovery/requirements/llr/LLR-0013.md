## Description

`kdr.quant.interface.QuantBackend` is a `typing.Protocol` declaring `apply_quant(model, quant_block: QuantBlockSubset) -> None`, `save(model, output_dir: Path) -> None`, and `name: str`. `QuantBlockSubset` is a typed Pydantic model with **flat optional fields** `weight: Optional[WeightQuantSpec] = None`, `key: Optional[KVQuantSpec] = None`, `value: Optional[KVQuantSpec] = None`. The backend acts only on populated (non-None) fields. The flat shape (rather than nested `kv_quant.key`/`kv_quant.value`) lets `factory.py` build per-quantizer sub-blocks without having to construct intermediate `kv_quant` containers.

## Rationale

Single `apply_quant` method (rather than `apply_weight_quant` + `apply_kv_quant`) preserves the "exactly one `mtq.quantize` call" invariant from HLR-0002 AC #3 without sacrificing the ability to mix backends per quantizer â€” the factory partitions the YAML's quant_block and dispatches a flat sub-block to each backend. Decouples the training loop from any specific quantization library.

The flat field shape is intentional and asymmetric to the YAML's nested `quant.kv_quant.{key,value}` structure: the YAML mirrors how humans think about the recipe (K and V belong together under "kv"), while the runtime sub-block flattens them so the backend sees three peers and can act on each independently.

## Acceptance Criteria

- [ ] Protocol declares exactly two methods (`apply_quant`, `save`) plus a `name: str` attribute
- [ ] `QuantBlockSubset` is a Pydantic v2 strict model (`ConfigDict(strict=True, extra='forbid')`) with EXACTLY three flat optional fields named `weight`, `key`, `value` (no nested `kv_quant` container)
- [ ] A backend that receives a `QuantBlockSubset` with all three fields == None raises `ValueError("backend received an empty quant block")`
- [ ] `factory.py`'s partition-or-merge logic produces ONE `QuantBlockSubset` per backend, not per quantizer. When `feature_matrix` routes the three quantizers to two backends (e.g. ModelOpt for weight, Native for K and V), the factory emits `QuantBlockSubset(weight=..., key=None, value=None)` to ModelOpt and `QuantBlockSubset(weight=None, key=..., value=...)` to Native â€” never two separate dispatches to the same backend. When all three quantizers route to ModelOpt the factory emits a single `QuantBlockSubset(weight=..., key=..., value=...)` so ModelOptBackend's `apply_quant` is still invoked exactly once
- [ ] mypy --strict accepts both `ModelOptBackend` and `NativeBackend` as conforming
