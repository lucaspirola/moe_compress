## Description

`kdr/configs/zaya1_8b_da_qad_profileJ_gguf.yaml` SHALL define the
Phase-7.2 Profile-J recovery recipe in `da_qad` mode against ZAYA1-8B
(self-distillation: `teacher.name_or_path` and `student.source` both
`Zyphra/ZAYA1-8B`). The YAML SHALL validate cleanly via
`kdr.config.Config.model_validate`.

The `quant.weight.spec_map` patterns SHALL target ZAYA1's *actual*
module substrings (verified via CPU dry-run on `Zyphra/ZAYA1-8B`
using the `lucaspirola/transformers@zaya1-patches` fork). Profile-J's
per-module-class bpw budget maps onto ZAYA1's fused-layout names as
follows:

| Module substring | ZAYA1 path(s) | Format | bits | granularity |
|---|---|---|---|---|
| `linear_fc1` | `…experts.local_experts.K.linear_fc1` (gate+up fused; 16 experts × 80 layers) | iq2_xs | 2 | block |
| `linear_fc2` | `…experts.local_experts.K.linear_fc2` (down projection; 16 × 80) | q3_k | 3 | block |
| `linear_q` | `…self_attn.qkv.linear_q` (attention query) | iq4_xs | 4 | block |
| `linear_k` | `…self_attn.qkv.linear_k` (attention key) | iq4_xs | 4 | block |
| `o_proj` | `…self_attn.o_proj` (attention output) | iq4_xs | 4 | block |
| `embed_tokens` | `model.embed_tokens` (tied with `lm_head` per ZAYA1's `tie_word_embeddings=True`) | q5_k | 5 | block |
| `lm_head` | `lm_head` | q5_k | 5 | block |

Modules NOT covered by these patterns — CCA value projections
(`qkv.val_proj1`, `qkv.val_proj2`), routing softmax tree (`router`,
`router.down_proj`, `router.rmsnorm_eda`), norms — remain F16 via
the ZAYA1 adapter's `fp32_carve_outs` list (HLR-0015 / LLR-0057).

The `quant.kv_quant` block SHALL use KIVI-style INT4 K (per-channel)
and INT4 V (per-token); runtime KV switches to `q4_0`/`q4_0` via
llama.cpp flags at deployment time.

Non-quant configuration (teacher / student / calibration /
distillation / eval) SHALL closely mirror
`zaya1_8b_da_qad_nvfp4_int4kv.yaml` — same hyper-params, same
hardware target.

## Rationale

Profile-J's original module-pattern table assumed canonical HF naming
(`gate_proj`, `up_proj`, `down_proj`, `q_proj`/`k_proj`/`v_proj`/
`o_proj`). ZAYA1 uses Megatron-style fused names instead. The
per-module-class bpw intent of Profile-J carries over directly:

- Gate+up fused into one Linear (`linear_fc1`) at 2× the width of an
  unfused gate or up — the same per-element 2.3 bpw applies; tensor
  size is 2× but the format budget is unchanged.
- Down (`linear_fc2`) at Q3_K matches the original `down_proj` budget.
- Attention split: ZAYA1 has only `linear_q` and `linear_k` as
  Linears (V is a fused-shared path in the CCA mechanism, hence
  `val_proj1`/`val_proj2`). Per Phase-7.2 design choice (user-locked
  this round): CCA values stay F16 via the carve-out; `linear_q` and
  `linear_k` get IQ4_XS like the original q/k/v/o.
- `o_proj` is the only attention path that retained its canonical
  name in the Zyphra fork; it stays at IQ4_XS.
- Embed and `lm_head` share storage (`tie_word_embeddings=True`); one
  Q5_K group with two `targets` patterns ensures both names route to
  the same recipe.

The patterns are chosen as the *smallest* substrings that
uniquely identify their targets. `linear_fc1` is unambiguous (no
other module name contains it), as is `linear_fc2`. `linear_q` /
`linear_k` are unique within ZAYA1's module list — they don't
collide with `linear_fc{1,2}` because `linear_q` is not a substring
of `linear_fc1`. `val_proj` substring (the CCA carve-out from
HLR-0015) does NOT appear in any `spec_map` pattern, so the carve-
out path applies to all CCA value Linears.

      a `Config` without raising `ValidationError`

        "o_proj", "embed_tokens", "lm_head"}`

      locked table above

      `(bits=4, format='int', granularity='channel', transform='none')`

      `(bits=4, format='int', granularity='token', transform='none')`

      `norm` (those are carve-outs, HLR-0015)
- [ ] `config.student.source == "Zyphra/ZAYA1-reasoning-base"`
      (self-distillation)
