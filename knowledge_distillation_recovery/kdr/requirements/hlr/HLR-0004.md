## Description

kdr SHALL define a `QuantBackend` Protocol with the methods `apply_quant(model, quant_block)` and `save(model, output_dir)`, where `quant_block` is the typed Pydantic config covering whichever subset of (weight, K, V) the backend has been routed to handle. A `ModelOptBackend` (default) SHALL implement the Protocol by translating its assigned subset of typed specs into a single modelopt config dict and invoking `mtq.quantize` EXACTLY ONCE per call; its `save` SHALL emit HF compressed-tensors safetensors via the converter selected by `feature_matrix` for the actual format used. A `NativeBackend` SHALL implement the Protocol with hand-rolled STE simulators that fill formats modelopt does not natively support (specifically INT3, INT2, and MXFP4-KV if absent in the installed modelopt version), using PyTorch forward hooks at adapter-declared module paths — Native's `apply_quant` therefore does NOT call `mtq.quantize`. A `feature_matrix.py` declares which `(target, format, bits)` tuples each backend supports; a `factory.py` partitions the YAML's `quant_block` per quantizer (weight, K, V), routes each part to the backend feature_matrix selects, and invokes each backend's `apply_quant` with a tailored sub-block. This design preserves "exactly one `mtq.quantize` call" (HLR-0002 AC #3) while still allowing Native to fill modelopt-shaped gaps on K or V independently of weight quantization.

## Rationale

Modelopt's API surface evolves quickly (NVFP4 KV, affine variants, rotation variants all landed in 2024-2026). A Protocol seam isolates kdr from those changes — bumping modelopt only touches `config_map.py` and `feature_matrix.py`, never the training loop. A Native fallback is necessary because modelopt cannot be assumed to ship every format we want to simulate; INT3/INT2 for K/V are common in low-bit research and modelopt's INT KV defaults are INT8/INT4 only at the time of writing. The Protocol form is "pragmatic seam" — it does NOT abstract over modelopt's internal wildcards or naming conventions, only over the operations kdr needs.

## Acceptance Criteria

- [ ] `kdr.quant.interface.QuantBackend` is a `typing.Protocol` with `apply_quant(model, quant_block) -> None` and `save(model, output_dir) -> None`; mypy --strict accepts both backends as conforming
- [ ] `ModelOptBackend.apply_quant` calls `mtq.quantize(model, modelopt_config_dict, calibrate_forward_loop)` exactly once per call
- [ ] `NativeBackend.apply_quant` does NOT call `mtq.quantize`; it installs PyTorch forward hooks at adapter-declared module paths
- [ ] `NativeBackend` implements STE for INT3 and INT2 with hypothesis-tested round-trip and gradient-flow properties
- [ ] `feature_matrix.py` is a const dict; updating modelopt requires only adding rows, never deleting Python code in the training loop
- [ ] `factory.py` partitions the YAML's `quant_block` per quantizer (weight, K, V) and routes each to the backend `feature_matrix` selects; the same YAML can route `weight` to ModelOpt and `kv_quant.value` to Native (or any other combination) without code changes
- [ ] Total `mtq.quantize` invocations during a single `da_qad` run is exactly 1 (when ModelOpt handles any quantizer) or 0 (when only Native is used) — never more — verified by a unit test that counts calls
- [ ] The training loop only references the `QuantBackend` Protocol via the factory — never modelopt directly
