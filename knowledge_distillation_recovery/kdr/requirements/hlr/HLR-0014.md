## Description

`save_kdr_artifact` SHALL produce a HF-loadable artifact for a Profile-J
recovery run whose `QuantBlock.weight` is a `MixedWeightSpec`. The
artifact is the compressed-tensors safetensors directory plus a
`config.json` whose `quantization_config` block reflects every distinct
quantization recipe in the spec_map.

Concretely:

1. `_build_quantization_config` SHALL emit one entry in `config_groups`
   per **unique `(format, bits, granularity)` triple** present in the
   spec_map. Each entry's `targets` list SHALL be the union of patterns
   whose triple landed in that group, preserving spec_map declaration
   order across groups.
2. The Uniform path (existing single-spec YAML) SHALL continue to emit
   exactly one `config_groups.group_0` entry with `targets: ["Linear"]`
   — unchanged from the v0 contract.
3. `NativeBackend.save` SHALL no longer raise `NotImplementedError`
   when the dispatched subset carries weight. For a Native-owned save,
   it SHALL emit a safetensors directory containing the
   simulated-quanted weights in BF16 storage (the parametrized
   `forward(weight)` already returns the fake-quanted tensor; saving
   reads the parametrized attribute and stores it as a plain BF16
   tensor). The per-pattern bit-widths and formats are NOT encoded into
   the safetensors layout itself — they are recorded by the caller's
   `_inject_quantization_config` step into `config.json`, and consumed
   downstream by the GGUF converter (Task 9 / HLR-0017).
4. `_build_quantization_config` MUST be the single point that
   translates a `WeightPatternSpec` (or `UniformWeightSpec`) into the
   compressed-tensors group dict — no duplicated mapping logic in
   `NativeBackend.save`.

The `ignore` list (FP32 carve-outs) and `kv_cache_scheme` blocks remain
on the same shape as v0 — no change to LLR-0020 in those slots.

## Rationale

llama.cpp's GGUF converter (Task 9) needs to recover the per-pattern
recipe from a Profile-J kdr artifact. The most ergonomic encoding is
one `config_groups` entry per `(format, bits, granularity)` triple
with the patterns listed under `targets`: the converter can iterate
groups and match tensor names against `targets` to decide which GGUF
encoder to invoke. Grouping by triple (rather than one group per
spec_map entry) collapses recipe-equivalent patterns into a single
declaration, matching how compressed-tensors itself groups parameters.

NativeBackend.save lifting `NotImplementedError` is required because
Profile-J's weight quant (all GGUF) lands entirely on Native — ModelOpt
cannot save it. The safetensors format itself doesn't change (it's
still bf16 tensors); the only new piece is that the parametrized
`weight` attribute is what gets serialized, capturing the
simulated-quant result.

## Acceptance Criteria

- [ ] For a Profile-J `MixedWeightSpec` with four distinct
      `(format, bits, granularity)` triples (e.g., IQ2_XS-2-block,
      Q3_K-3-block, IQ4_XS-4-block, Q5_K-5-block),
      `_build_quantization_config` produces a `config_groups` dict with
      exactly four entries
- [ ] For a `MixedWeightSpec` whose two entries share the same
      `(format, bits, granularity)` triple but distinct patterns,
      `_build_quantization_config` produces a single `config_groups`
      entry whose `targets` list contains both patterns in declaration
      order
- [ ] The Uniform-config save path is unchanged: a YAML with a single
      `UniformWeightSpec` produces one `config_groups.group_0` entry
      with `targets: ["Linear"]`
- [ ] `_build_quantization_config` raises `ValueError` (not
      `NotImplementedError`) if both branches of the
      `UniformWeightSpec | MixedWeightSpec` union are exhausted (the
      Pydantic validator forbids construction of any third branch, so
      this is defense-in-depth — a typed exhaustiveness check)
- [ ] `NativeBackend.save` no longer raises `NotImplementedError`
      when the dispatched subset carries weight; instead it writes
      safetensors via `unwrapped.save_pretrained(output_dir,
      state_dict=..., safe_serialization=True)` with BF16 storage
- [ ] The state_dict passed to `save_pretrained` contains the
      parametrized weight tensors (i.e., reading `module.weight` while
      parametrization is active produces the fake-quanted tensor,
      which `save_pretrained` then serializes verbatim)
- [ ] An end-to-end test on a stand-in model with a 2-entry
      `MixedWeightSpec` confirms the resulting `config.json`'s
      `quantization_config.config_groups` has the expected two entries
      with the expected `targets`
- [ ] An existing Uniform-path save test continues to pass without
      modification — back-compat preserved
- [ ] `ignore: list(fp32_carve_outs)` and the `kv_cache_scheme` block
      shape are unchanged
