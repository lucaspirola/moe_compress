## Description

The ZAYA1 adapter SHALL populate `fp32_carve_outs` with substring
patterns that match every ZAYA1-specific module that must remain at
F16 during `da_qad` recovery, irrespective of which weight recipe
the YAML specifies. The substrings SHALL be chosen against ZAYA1's
*actual* module layout (verified by a CPU dry-run on
`Zyphra/ZAYA1-8B`, confirming names exposed by the
`lucaspirola/transformers@zaya1-patches` fork), not against
canonical-HF naming conventions.

The carve-out set is:

| Substring | Targets |
|---|---|
| `lm_head` | `lm_head` (matmul; LLR-0024 AC #1) |
| `embed_tokens` | `model.embed_tokens` (input embedding; tied with `lm_head` per `tie_word_embeddings=True`) |
| `router` | `model.layers.N.zaya_block.router` and its internal `router.down_proj` / `router.rmsnorm_eda` â€” the routing softmax module tree |
| `norm` | `input_norm`, `post_attention_layernorm`, RMSNorm variants |
| `val_proj` | `self_attn.qkv.val_proj1` and `self_attn.qkv.val_proj2` â€” the CCA value projections (ZAYA1's CCA "downprojector" path) |

Profile-J's `spec_map` MAY override `lm_head` and `embed_tokens` via
explicit patterns (the locked precedence: explicit pattern wins over
carve-out, per HLR-0013). All other carve-out entries SHALL remain
F16 â€” no Profile-J pattern should target them.

The `cca_cache` substring (Phase 7.2 pre-launch hypothesis) is
removed: CCA cache state is a runtime tensor buffer, not an
`nn.Module`, so substring-matching against `named_modules()` never
matches it. Removing it from the list is cleanup; the
`fp32_carve_outs` mechanism cannot affect runtime buffers anyway.

The `cca_conv` substring (Phase 7.2 pre-launch hypothesis) is
replaced by `val_proj`: the CCA downprojector in ZAYA1's actual fork
is implemented as the value-projection Linears
`self_attn.qkv.val_proj{1,2}`, not as a `cca_conv` module.

## Rationale

Profile-J's bpw budget allows the attention QKVO at IQ4_XS, but the
CCA value projections sit at the head of the attention output path
and are highly sensitive to quantization noise (small parameter
count, high gradient throughput). The conservative choice â€” also
the one the original Profile-J intent encoded as "cca_conv stays
F16" â€” is to keep them at F16. The substring `val_proj` matches both
`val_proj1` and `val_proj2` (and nothing else in the inspected
ZAYA1 module list).

The `router` substring is deliberately broad: it matches the
routing softmax module path AND its internal `router.down_proj`
Linear AND `router.rmsnorm_eda`. All three are routing-related and
F16-sensitive. The locked precedence (explicit pattern wins over
carve-out) gives YAML authors an escape hatch if they ever want to
quantize a specific router-internal Linear.

## Acceptance Criteria

- [ ] `Zaya1Adapter.fp32_carve_outs(model)` returns a list whose
      contents (order-independent) equal
      `["lm_head", "embed_tokens", "router", "norm", "val_proj"]`
- [ ] The list has exactly 5 entries â€” no duplicates, no fictional
      `cca_conv` / `cca_cache`
- [ ] On the real ZAYA1-8B model (or a fixture with matching module
      names), each substring resolves to at least one `nn.Module`
- [ ] `"val_proj"` matches both `qkv.val_proj1` and `qkv.val_proj2`
- [ ] `"val_proj"` does not collide with any other ZAYA1 module name
      (no Linear named `val_proj` outside the CCA path)
- [ ] Existing tests covering the v0 carve-out behaviour
      (`test_fp32_carve_outs_includes_lm_head`,
      `test_fp32_carve_outs_resolve_to_non_empty_submodule_sets`)
      continue to pass against the updated list
