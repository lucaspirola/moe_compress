## Description

`kdr.quant.native_backend.gguf_codebooks.get_iq2xs_grid` and
`get_ksigns_iq2xs` SHALL construct their unpacked codebook tensors
using GPU-native tensor operations when the target device is a GPU,
rather than building a Python list on CPU and then transferring to
the target device.

The functions are cache-miss-only (once per `(device, dtype)` pair
per process lifetime) so the runtime impact is negligible — this is
a cleanliness/best-practice cleanup flagged by Finding 3 of the
pipeline CPU→GPU audit.

The unpacking SHALL be bit-identical to the existing Python-loop
implementation: byte-extraction from uint64 (for `get_iq2xs_grid`)
and bit-extraction from uint8 (for `get_ksigns_iq2xs`) are
arithmetic operations that produce identical integer outputs whether
done in Python or in PyTorch tensor ops.

## Rationale

Two reasons to clean this up despite its low runtime impact:

1. The codebook constants are stored as `tuple[int, ...]` Python
   tuples; the current implementation materialises a Python list,
   then `torch.tensor(list, device=device)` does a host allocation
   and an H2D copy. Doing the unpack with `torch.tensor(raw,
   device=device)` plus bitwise tensor ops keeps the work on-device
   from the start, mirroring the discipline applied to the rest of
   the STE simulator code in `ste_simulators.py`.

2. The Python loops are the only nested-loop constructs in
   `gguf_codebooks.py`; replacing them with vectorised tensor ops
   improves readability and removes a "code-smell" pattern from a
   module that otherwise houses only constants + cache lookups.

## Acceptance Criteria

- [ ] `get_iq2xs_grid(device, dtype)` returns a tensor whose values
      are bit-identical to the prior Python-loop implementation for
      every (device, dtype) it accepts
- [ ] `get_ksigns_iq2xs(device, dtype)` likewise
- [ ] The cache contract is unchanged: repeated calls with the same
      `(device, dtype)` return the same tensor object (identity, not
      equality)
- [ ] The implementations no longer contain a Python-level
      `for val in IQ2XS_GRID_RAW: for i in range(8): ...` nested
      loop pattern; the unpack uses vectorised PyTorch ops
- [ ] A pytest test asserts the new implementations'
      `torch.equal`-bit-identity against an inline reference replay
      of the old Python-loop implementation, for each function and
      for at least one `(device, dtype)` representative pair (CPU +
      meta + a float dtype)
- [ ] Existing tests in `tests/property/test_ste_simulators.py`
      (which exercise `get_iq2xs_grid` / `get_ksigns_iq2xs` via the
      STE simulators) continue to pass
- [ ] No new `# type: ignore` comments
