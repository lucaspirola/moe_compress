## Description

A standalone CLI tool `kdr/tools/kdr_to_gguf.py` SHALL convert a saved
kdr artifact (the compressed-tensors safetensors directory produced by
`save_kdr_artifact`) into a single `.gguf` file loadable by llama.cpp.

The tool SHALL:

1. Accept exactly two required CLI arguments:
   `--kdr-dir <path>` (input directory) and `--output <path.gguf>`
   (output file path). Any additional flags are optional and do not
   change the core contract.
2. Read `<kdr-dir>/config.json` to recover both the HF model hyper-
   parameters (`hidden_size`, `vocab_size`, etc.) and the
   `quantization_config` block emitted by `_build_quantization_config`
   (LLR-0056). The `config_groups` field SHALL drive per-tensor
   encoder selection.
3. For each tensor in the safetensors files under `<kdr-dir>/`:
   - identify which `config_groups.group_N.targets` pattern (if any)
     matches the tensor's name (substring match, same semantics as
     `NativeBackend._is_carved_out`);
   - if a pattern matches, encode the tensor using the GGUF encoder
     for that group's `(format, bits)` triple (IQ2_XS, Q3_K, IQ4_XS,
     Q5_K for Profile-J);
   - if no pattern matches AND the tensor's name matches the
     `quantization_config.ignore` list (FP32 carve-outs), encode the
     tensor as F16 (the GGUF fallback type);
   - if neither matches: raise `RuntimeError` naming the orphan
     tensor (no silent FP16 fallback — locked design choice #5).
4. Handle tied `embed_tokens` / `lm_head` correctly. ZAYA1 stores
   them as one shared tensor on disk; the tool SHALL encode it once
   and write it under the GGUF convention name expected by llama.cpp
   (`token_embd.weight`). llama.cpp's loader resolves the tie at load
   time.
5. Write a valid GGUF file with header, metadata, and the encoded
   tensor table. The metadata SHALL include the architecture name,
   the hparams from `config.json`, and a kdr-provenance string that
   records the kdr version and the source `--kdr-dir`.
6. Use the `gguf` Python library (llama.cpp's official writer package,
   installable via `pip install gguf`) as a HARD dependency. The tool
   SHALL exit cleanly with an actionable error message if `gguf` is
   not installed, rather than crashing on import.

The tool runs OFFLINE on a CPU; no GPU or distributed setup is
required. It does NOT perform any forward pass on the model —
encoding is purely tensor-to-tensor.

## Rationale

llama.cpp's GGUF reference is the authoritative serialisation format
for the deployment target (GTX 1050 Mobile). Using llama.cpp's own
`gguf` Python library guarantees byte-level compatibility without
reimplementing 4 encoder paths from scratch (~thousands of LOC) and
without diverging from upstream when ggml updates a codebook layout
(the library tracks the upstream commit history).

The tool reads from `config.json::quantization_config` (not from the
YAML) because the YAML may not be present at conversion time (kdr
artifacts are self-describing). The same `config_groups` structure
that compressed-tensors emits is exactly what the converter consumes.

The eager `RuntimeError` on unmatched tensors mirrors LLR-0013's
no-silent-FP16-fallback contract; it ensures conversion fails loudly
when an unforeseen tensor appears (e.g., a new layer added after
training that wasn't in the spec_map or carve-outs).

## Acceptance Criteria

- [ ] `kdr/tools/kdr_to_gguf.py` exists and is executable as a script
      (`python -m kdr.tools.kdr_to_gguf --kdr-dir ... --output ...`
      OR `python kdr/tools/kdr_to_gguf.py --kdr-dir ... --output ...`)
- [ ] The CLI accepts `--kdr-dir <path>` and `--output <path.gguf>`
      as required arguments; missing either prints a usage error
- [ ] When `gguf` is not installed, `import` fails fast at module
      load time with a clear `ModuleNotFoundError` or an
      `ImportError` whose message mentions `pip install gguf`
- [ ] For a stand-in kdr-dir with a config.json having a 2-group
      `config_groups` (one IQ2_XS pattern, one F16 carve-out), the
      tool produces a valid `.gguf` file that opens via
      `gguf.GGUFReader` without raising
- [ ] On a stand-in kdr-dir whose safetensors contains a tensor
      named `mystery_tensor` not covered by any `config_groups.targets`
      pattern AND not listed in `quantization_config.ignore`, the
      tool raises `RuntimeError` whose message contains the literal
      string `mystery_tensor`
- [ ] The tool's GGUF output includes a metadata key
      `general.architecture` whose value derives from the input
      `config.json::architectures[0]` via lowercase + suffix-strip
      (e.g., `"ZayaForCausalLM"` maps to `"zaya"`; `"LlamaForCausalLM"`
      maps to `"llama"`)
- [ ] The tool's GGUF output includes a metadata key
      `kdr.source_dir` whose value is the absolute path of the
      input `--kdr-dir`
- [ ] A tied embed/lm_head input (one shared tensor in safetensors)
      produces exactly one `token_embd.weight` tensor in the GGUF
      output (not two)
- [ ] No `# type: ignore` comments other than for the `gguf` library
      import (which has incomplete stubs)
