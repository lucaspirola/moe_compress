## Description

`kdr.quant.native_backend.ste_simulators._iq2xs_snap_block` SHALL
size its argmin-loop tile such that the Python-level loop dispatches
≤ O(few hundred) CUDA kernels per weight matrix of the shapes
encountered in Profile-J `da_qad` training, while keeping the
in-flight `inner`/`dist` intermediates under a documented memory
budget (default ≤ 4 GB).

The function's **numerical output** SHALL be bit-identical to the
existing implementation: the tile-loop refactor is purely an
amortisation of CUDA kernel launch overhead, not a change in the
quant algorithm.

Rationale: with the current `_IQ2XS_ARGMIN_TILE = 64`, a 4096×4096
Linear weight (2M chunks of 8 elements) dispatches ~32 000 separate
kernel launches per IQ2_XS forward pass. On Profile-J ZAYA1-8B with
~16 MoE experts × 80 layers + attention QKV + tied embed_tokens/
lm_head (~200 IQ2_XS-applicable weight matrices), each forward
issues ~6.4 M Python→CUDA dispatcher crossings. With ~5–10 μs of
kernel-launch overhead per crossing on H200/B200, that's
~30–60 seconds of pure overhead per training-step forward, on top
of the actual matmul time.

A larger tile (e.g., 8192) reduces this to ~250 iterations per
weight matrix, ~50 000 total dispatcher crossings per forward —
two orders of magnitude lower overhead at the same memory footprint
profile (≤4 GB intermediate, which fits comfortably even after the
17 GB+ teacher and 17 GB+ student weights are resident on a 141–180
GB GPU).

## Acceptance Criteria

- [ ] `_IQ2XS_ARGMIN_TILE` (or a new shape-adaptive equivalent) is
      sized so that for any input where `n_total ≤ 8192`, the loop
      runs as a single iteration (no Python-level tiling overhead)
- [ ] For larger inputs, the tile size is bounded such that the
      `inner` and `dist` intermediates each fit within a documented
      ≤4 GB budget per tile, and the loop iteration count is at
      most `ceil(n_total / 8192)` for typical Profile-J weight
      shapes
- [ ] The function's output is bit-identical to the prior
      implementation for representative inputs (≥1 test asserts
      `torch.equal(new_output, old_output)` on a curated input set
      with last-axis size that is a multiple of 256)
- [ ] No new `# type: ignore` comments introduced
- [ ] All existing `tests/property/test_native_parametrizations.py`
      and `tests/property/test_ste_simulators.py` tests continue to
      pass — they already exercise STE gradient identity and
      property invariants
- [ ] A new property test verifies the tile-refactor preserves the
      output across multiple representative shapes (e.g., last-axis
      = 256, 512, 4096, 8192)
- [ ] The maximum tile is documented in a comment with the per-tile
      memory budget arithmetic (so a future reviewer can recompute
      and adjust if the budget needs to change)
