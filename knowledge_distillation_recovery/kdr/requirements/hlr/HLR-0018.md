## Description

For any `da_qad`-mode run, the student model SHALL be on the trainer's
target device (`accelerator.device`) BEFORE
`kdr.quant.factory.partition_and_dispatch` is invoked. The
`calibrate_loop` closure consumed by ModelOpt's `mtq.quantize` SHALL
move each calibration batch to the model's device before forward.

This is required because `mtq.quantize` invokes `calibrate_forward_loop`
synchronously during the call: any GPU acceleration of the PTQ
calibration is determined by the device the model is on at that moment.
A model on CPU during apply_quant runs the entire calibration on CPU â€”
unusably slow for 8B+ models.

Constraints to preserve:

1. **LLR-0048's locked ordering** â€” `mtq.quantize` MUST occur INSIDE
   the `activate_zero3_init` context manager and BEFORE
   `accelerator.prepare`. The fix SHALL NOT move
   `partition_and_dispatch` outside the context or after
   `accelerator.prepare`.
2. **DeepSpeed ZeRO-3 path** â€” under ZeRO-3, the model is
   partitioned across ranks by `activate_zero3_init`; an explicit
   `.to(device)` would break the partitioning. The fix SHALL skip
   the device move under DeepSpeed.
3. **DDP / single-GPU non-DS path** â€” `accelerator.prepare` (Stage 2)
   is the canonical device-placement step. The fix SHALL move the
   student device-ward at Stage 1 (post-load, pre-dispatch) AS A
   PRECONDITION for `apply_quant`, leaving `accelerator.prepare`'s
   DDP-wrap responsibility intact.

## Rationale

Phase 7.2 launch on a Datacrunch B200 confirmed this issue live:
trainer ran for 9+ min post-`apply_quant` with VRAM=4 MiB / 183 GiB
and GPU utilisation 0%, while a single Python process pegged 15 CPU
cores. `mtq.quantize`'s 256-sequence calibration was running on CPU
end-to-end. Without the fix, every single-GPU `da_qad` run on real
8B+ models is effectively bricked at calibration.

The two-place fix mirrors the two-place root cause: (a) the model
isn't moved before `partition_and_dispatch`; (b) even if it were, the
calibrate_loop hands CPU tensors to the model. Fixing (a) without (b)
produces a device-mismatch crash; fixing (b) without (a) keeps
calibration on CPU. Both must land.

## Acceptance Criteria

- [ ] `kdr.training.loop.run_recovery` moves `student` to
      `accelerator.device` after `adapter.load_teacher_and_student`
      returns and before `partition_and_dispatch` is invoked, under
      the condition `not is_deepspeed(accelerator)`. The move sits
      INSIDE the `activate_zero3_init` context.
- [ ] When `is_deepspeed(accelerator)` is True, the explicit move
      is skipped (ZeRO-3 partitioning is honored).
- [ ] `partition_and_dispatch`'s call signature is unchanged â€” the
      caller (Stage 1) places the model; the factory takes whatever
      device the caller hands it.
- [ ] `kdr.quant.factory._make_calibrate_loop`'s returned closure
      moves each batch to `next(model.parameters()).device` before
      calling `model(input_ids=batch)`. This makes the loop robust
      regardless of whether the model is on CPU, on a single GPU, or
      partitioned (in which case `next(model.parameters()).device`
      points to whichever shard's parameter is local).
- [ ] On a non-DeepSpeed test that mocks `mtq.quantize` and inspects
      the model passed in, the model is observably on the
      accelerator's device.
- [ ] On a test that runs the `calibrate_loop` closure against a
      CUDA-shaped MagicMock model with `parameters()` returning a
      CUDA-device tensor, every batch handed to the model's forward
      is on the same device as the model.
- [ ] `accelerator.prepare`'s Stage-2 invocation is unchanged in
      shape (still receives student + optim).
- [ ] The Stage-3 teacher `.to(accelerator.device)` is unchanged.
- [ ] All existing tests pass; no regression in the DeepSpeed code
      path (verified by `is_deepspeed` branch coverage).
