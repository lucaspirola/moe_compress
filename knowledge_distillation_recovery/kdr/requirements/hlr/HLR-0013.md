## Description

NativeBackend SHALL install `nn.utils.parametrize` parametrizations on
every targeted `nn.Linear` module such that — for any `apply_quant` call
carrying a `QuantBlockSubset` whose `weight` is a non-empty
`list[WeightPatternSpec]` — each Linear's `weight` tensor is fake-quanted
according to the first matching `WeightPatternSpec` from the routed
list, with STE gradient pass-through.

The installation logic SHALL implement the following precedence per
`nn.Linear` module, in this order:

1. **Explicit (non-empty) pattern match** wins. If any
   `WeightPatternSpec` whose `pattern` is non-empty has its `pattern`
   substring in the module's qualified name, the first such spec
   (in declaration order) is installed. This binds explicit
   per-pattern Profile-J entries over both carve-outs and fallback
   defaults — locked design choice #5.
2. **Carve-out** wins. If no explicit pattern matched AND the module's
   qualified name matches a `fp32_carve_outs` entry, the module is
   left untouched.
3. **Empty-pattern fallback** applies. If no explicit pattern matched
   AND no carve-out matched, but the spec_map contains an entry with
   `pattern=""`, that entry is installed (the uniform-config shim
   path; one-entry `spec_map` with `pattern=""` is how v0 uniform YAMLs
   are normalised by `_normalize_weight_to_patterns`).
4. **Error** otherwise. If nothing above applied, raise `ValueError`
   naming the unmatched module. Silent FP16 fallback is forbidden.

Format dispatch (independent of precedence): for any spec that
reaches installation, the parametrization corresponding to that spec's
`format` is installed. Supported formats in Phase 7.2: `int`,
`iq2_xs`, `q3_k`, `iq4_xs`, `q5_k`. All other GGUF formats accepted
by LLR-0010's `Format` Literal SHALL raise `NotImplementedError` at
installation time (not at parametrization-forward time) so that a
malformed config fails eagerly.

Existing uniform-config behaviour MUST be preserved: a one-entry
spec_map with `pattern=""` installs on every non-carve-out Linear
identically to the v0 path — the carve-out-then-fallback precedence
above is exactly what makes that work.

Granularity values incompatible with the chosen format SHALL raise
`NotImplementedError` at installation time, so that config-format
incoherence fails eagerly alongside other malformed-config paths.

## Rationale

Profile-J recovery's correctness depends on every Linear getting the
right codebook STE applied. A first-match-wins precedence over the
spec_map gives users a predictable rule (declaration order = priority)
without introducing a separate priority field. Raising eagerly on
unmatched modules surfaces config drift (renamed layers, fork
divergence) at training-start rather than letting an FP16 layer
silently leak into a quantized recipe.

The four supported GGUF formats correspond to the four STE simulators
shipped under LLR-0015. Other GGUF formats are accepted by the schema
(LLR-0010) but have no simulator — failing eagerly at apply_quant time
keeps the error path tight to the malformed config rather than
deferring to a runtime forward.

## Acceptance Criteria

- [ ] `NativeBackend.apply_quant` accepts a `QuantBlockSubset` whose
      `weight` is a `list[WeightPatternSpec]` of any length ≥ 1 (the
      Task-3 stopgap multi-entry guard is removed)
- [ ] On a model where two Linear modules with distinct names both
      match the same pattern, both receive the parametrization
      corresponding to that pattern
- [ ] On a model with three Linears named `mlp.gate_proj`,
      `mlp.up_proj`, `mlp.down_proj`, and a spec_map of
      `[(pattern="gate_proj", fmt=iq2_xs), (pattern="up_proj", fmt=iq2_xs),
        (pattern="down_proj", fmt=q3_k)]`, all three install with the
      matching format-specific parametrization
- [ ] On a model with a Linear named `attn.q_proj` and a spec_map whose
      first matching entry is `(pattern="proj", fmt=iq4_xs)`,
      first-match-wins means `q_proj` receives IQ4_XS even if a later
      `(pattern="q_proj", fmt=q5_k)` entry exists
- [ ] On a model with a Linear named `model.lm_head` and a
      `fp32_carve_outs=["lm_head"]` carve-out, no parametrization is
      installed (carve-out applies after the spec_map miss)
- [ ] If a Linear named `weird_layer` matches no spec_map entry and no
      carve-out, `apply_quant` raises `ValueError` whose message
      contains the literal module name `weird_layer`
- [ ] A spec_map containing a `WeightPatternSpec` with
      `format='iq3_xs'` raises `NotImplementedError` at `apply_quant`
      time (eager fail; not deferred to forward)
- [ ] Uniform shim path: a one-entry spec_map with `pattern=""` still
      installs the v0 INT path on every non-carve-out Linear
- [ ] The implementation uses `nn.utils.parametrize.register_parametrization`
      (same mechanism as the existing INT path)
- [ ] Hypothesis property tests confirm the four GGUF parametrizations
      pass STE gradients identically to direct calls of the underlying
      simulators
- [ ] A `WeightPatternSpec` whose `granularity` value is incompatible
      with its `format` (e.g., a GGUF format paired with
      `granularity='channel'`, or `int` paired with `granularity='block'`)
      raises `NotImplementedError` at `apply_quant` installation time
