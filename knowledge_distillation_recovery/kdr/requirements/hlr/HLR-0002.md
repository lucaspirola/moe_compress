## Description

kdr SHALL expose a single `mode` enum with values `"bf16" | "da_qad"`. In `bf16` mode kdr runs Chapter-1 logit KD only — the student is loaded as a plain BF16 transformer and `mtq.quantize` is NOT invoked. In `da_qad` mode kdr runs Chapter-3 deployment-aware QAD — `mtq.quantize` IS invoked at the start of training, installing fake-quant wrappers per the YAML's `quant` block before any training step. Teacher topology, loss function, optimizer, eval, and save/resume contract SHALL be identical across modes.

## Rationale

A binary mode flag keeps the package surface small (one tool, one CLI, one config schema) while letting the same FKLD loop drive both Ch1 and Ch3. The flag is binary because every other knob (formats, granularities, layer-exempt indices) is already expressible in the YAML quant block — a finer-grained mode enum would only duplicate state. The "skip `mtq.quantize` in bf16" pattern is what makes `bf16` mode reduce exactly to the structural_recovery behavior the parity smoke (Phase 7.1) compares against.

## Acceptance Criteria

- [ ] `kdr.modes` exports a `Literal["bf16", "da_qad"]` type used by Pydantic config validation
- [ ] In `bf16` mode the trainer makes zero calls to `modelopt.torch.quantization.mtq.quantize`
- [ ] In `da_qad` mode the trainer makes exactly one `mtq.quantize` call before the first training step, with the modelopt config dict derived from the YAML's `quant` block
- [ ] Mode switching is config-only — no Python branches per mode beyond the single `mtq.quantize` gate
- [ ] An invalid mode value in the YAML raises a Pydantic ValidationError at config load, not at runtime
