## Description

`kdr.eval.quick._wikitext2_ppl` SHALL construct the WikiText-2 input-ids
tensor directly on `accelerator.device` rather than constructing it on
CPU and then issuing a separate host→device copy.

This is a cleanup of a CPU-bound pattern flagged by the Finding 2
audit of the pipeline: `torch.tensor(ids_list, dtype=torch.long).view(...).to(accelerator.device)`
allocates on CPU and copies, when `torch.tensor(ids_list, dtype=torch.long,
device=accelerator.device).view(...)` skips the CPU intermediate. The
two forms are numerically identical.

## Rationale

This is an off-hot-path cleanup — eval runs at `eval_every_n_steps`
cadence, not every training step. The win is small in wall-clock
terms but the pattern is gratuitous: there is no reason to pay for
a CPU allocation and an H2D copy when the constructor accepts a
`device` kwarg.

## Acceptance Criteria

- [ ] The single `torch.tensor(ids_list, dtype=torch.long)` call in
      `eval/quick.py` constructs with `device=accelerator.device`
      directly; no separate `.to(accelerator.device)` follows it
- [ ] The `.view(n_seqs, seq_len)` chain is preserved (shape is
      unchanged)
- [ ] Existing `tests/test_eval_quick.py` tests continue to pass
      without modification
- [ ] No new `# type: ignore` introduced
- [ ] A small unit test asserts the constructed tensor is on
      `accelerator.device` after the construction line (mock-driven,
      using `torch.device("meta")` to keep CPU-only test infra
      working)
