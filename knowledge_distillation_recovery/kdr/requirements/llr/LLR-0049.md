## Description

The YAML's `distillation` block SHALL include a required field `eval_every_n_steps: int` (>= 1). The training loop dispatches to `kdr.eval.quick._wikitext2_ppl(...)` every `eval_every_n_steps` optimizer-update boundaries and ONLY then. The dispatch happens AFTER the gradient step and AFTER any in-step logging, on rank 0 with collective forward semantics preserved (rank-replicated forward — every rank participates in the eval forward pass even though only rank 0 logs the result, per LLR-0036).

## Rationale

HLR-0009's AC #5 ("Eval PPL is logged exactly once per `eval_every_n_steps`") implicitly requires this field exist as a typed config and that the dispatch mechanism honours it. Without an explicit LLR for the field + dispatch logic, the AC is unverifiable.

## Acceptance Criteria

- [ ] `kdr.config.DistillationConfig` (the model containing this field along with `total_tokens`, `learning_rate`, etc.) declares `eval_every_n_steps: int` as a required, strictly-positive integer (Pydantic `Field(..., gt=0)`)
- [ ] Pydantic raises `ValidationError` on `eval_every_n_steps: 0` or negative values
- [ ] The training loop's dispatch fires when `step > 0 and (step % cfg.distillation.eval_every_n_steps) == 0`, AFTER the optimizer step (not before). **Step 0 does NOT trigger eval** — a step-0 baseline eval is a separate concern (out of scope for this LLR; if a baseline is ever wanted it is invoked once before the loop starts, not via this cadence)
- [ ] The dispatch is collective: every rank calls `_wikitext2_ppl` even though only rank 0 logs
- [ ] A unit test runs a fake training loop for 50 steps with `eval_every_n_steps=10` and asserts eval was invoked exactly 5 times (steps 10, 20, 30, 40, 50) — NOT 6 (no step-0 eval)
