## Description

`kdr.quant.specs` exposes a discriminated union of two Pydantic v2 strict
models for weight quantization, plus a pattern element type used by the
Mixed branch:

- `UniformWeightSpec` — the per-model weight spec. Required fields
  `bits`, `format`, `granularity`, `transform`; no defaults; `transform`
  accepts `Literal['none']` only in v0. `model_config = ConfigDict(strict=True, extra='forbid')`.
- `WeightPatternSpec` — same field set as `UniformWeightSpec` plus a
  required `pattern: str` (a substring matched against `nn.Linear`
  qualified names; `pattern=""` is the catch-all). All five fields are
  required, including `transform`.
- `MixedWeightSpec` — wraps `spec_map: list[WeightPatternSpec]` with
  `Field(..., min_length=1)` and a `_no_duplicate_patterns` validator
  (duplicate `pattern` strings are rejected). First-match-wins
  precedence within the list is mandated by LLR-0017.

`WeightQuantSpec` is preserved as a module-level alias of
`UniformWeightSpec` for one release so external imports keep working.

`kdr.config.QuantBlock.weight` is the union
`UniformWeightSpec | MixedWeightSpec`. Pydantic smart-mode discriminates
by the presence/absence of `spec_map`; no explicit discriminator field
is added.

The `Format` Literal enumerates the full llama.cpp / GGUF catalog
alongside the v0 set: `int`, `fp8`, `mxfp4`, `nvfp4`, `q4_0`, `q4_1`,
`q5_0`, `q5_1`, `q8_0`, `q2_k`, `q3_k`, `q4_k`, `q5_k`, `q6_k`, `q8_k`,
`iq1_s`, `iq1_m`, `iq2_xxs`, `iq2_xs`, `iq2_s`, `iq2_m`, `iq3_xxs`,
`iq3_xs`, `iq3_s`, `iq3_m`, `iq4_xs`, `iq4_nl` (27 entries). STE
implementations only ship for the four Profile-J formats today
(LLR-0015); the rest are accepted by the schema and raise
`NotImplementedError` at `apply_quant` time until claimed.

## Rationale

Same shape as `KVQuantSpec` to enable shared validation logic. Explicit
`extra='forbid'` ensures unknown YAML fields are rejected, not silently
ignored — Pydantic's `strict=True` alone does NOT imply `extra='forbid'`.

The Mixed branch is required by Profile-J recovery: per-module-pattern
quantization (IQ2_XS for `gate_proj/up_proj`, Q3_K for `down_proj`,
IQ4_XS for `q/k/v/o_proj`, Q5_K for `embed_tokens`/`lm_head`) cannot be
expressed by a single global weight spec. The union-with-alias approach
keeps every existing uniform YAML loading unchanged while the new Mixed
shape is parsed by the same `QuantBlock`.

The `Format` catalog is broader than the v0 STE coverage on purpose:
the schema is the contract that Profile-J YAMLs are written against, so
it must enumerate every GGUF target even though only four are
implemented today. Backend dispatch (LLR-0015) is responsible for the
deferred-STE error.

## Acceptance Criteria

- [ ] `Format` Literal contains all 27 entries above
- [ ] `UniformWeightSpec` is a Pydantic strict model
      (`extra='forbid', strict=True`) with four required fields
      (`bits`, `format`, `granularity`, `transform`) and no `pattern`
      field
- [ ] `WeightQuantSpec` is a module-level alias re-export of
      `UniformWeightSpec`; `from kdr.quant.specs import WeightQuantSpec`
      still resolves
- [ ] `WeightPatternSpec` is a strict model with required `pattern: str`
      plus the same four fields as `UniformWeightSpec`
- [ ] `MixedWeightSpec.spec_map` has `min_length=1`; an empty
      `spec_map` raises `ValidationError`
- [ ] `_no_duplicate_patterns` rejects a `spec_map` containing two
      entries with the same `pattern` string
- [ ] `kdr.config.QuantBlock.weight` parses both a uniform YAML
      (`weight: {bits, format, granularity, transform}`) and a mixed
      YAML (`weight: {spec_map: [{pattern, …}, …]}`); Pydantic
      discriminates smart-mode without an explicit discriminator
- [ ] Symmetric 4-bit weights are expressible
      (`bits=4, format='int', granularity='channel', transform='none'`)
- [ ] Pydantic raises `ValidationError` on a YAML containing an unknown
      field (e.g., `nbits` instead of `bits`)
- [ ] mypy --strict accepts the union and the per-branch field access
