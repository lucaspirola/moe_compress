## Description

On every `save_every_n_steps` boundary, the trainer writes a partial
checkpoint to `artifacts_dir / f'kdr_{mode}_partial_step{step}'` using the
same `save_kdr_artifact` path used for final saves.

The disk-write phase of `save_partial` MAY run in a background thread when
`distillation.enable_async_save: true` is set in the trainer config. Sync
remains the default for backward compatibility and ease of debugging. The
collective consolidation phase (`accelerator.get_state_dict`) always runs
synchronously on the trainer's main thread because NCCL is not thread-safe;
only the rank-0 disk I/O (save_pretrained, tokenizer write, metadata, atomic
rename, sentinel) is dispatched to the background.

Async semantics (when enabled):

- Single-flight queue, depth 1. The next `save_partial(async_mode=True)`
  call auto-joins the prior pending write before submitting a new one.
  Guarantees monotone partial ordering on disk and on HF Hub — no race
  where partial_step{N+10} writes before partial_step{N}.
- `save_partial_join()` is callable explicitly (e.g., before the final
  save) to flush any pending writes; any exception raised in the
  background thread re-raises at the join site so failures fail fast
  rather than silently dropping a partial.
- `async_mode=True` is rejected for `partial=False` (the final save):
  the caller must `save_partial_join()` first and then run the final
  save synchronously, because its return value (the final path) must be
  a fully-committed dir.
- The CPU `state_dict` snapshot is deep-copied off the main thread to
  guarantee subsequent optimizer.step() and ZeRO-3 internal-buffer churn
  cannot mutate the snapshot while the background thread serializes it.

## Rationale

Phase 7.1 observed that the rank-0 disk-write phase (state_dict
serialization + tokenizer + metadata writes + atomic rename + sentinel)
stalls training for ~10-30 s per partial on an 8B model — a meaningful
fraction of total wall-time at `save_every_n_steps=10` cadence. Moving
this phase off the GPU-critical path overlaps it with the next training
microbatches.

The async path is opt-in (not default) for two reasons: (a) sync is
easier to debug when partials produce unexpected content, and (b) the
extra CPU pinned-memory cost (~weight-tensor size — for an 8B BF16
model ~17 GB, for a 70B BF16 model ~140 GB) is a real cost the user
must consciously accept on their host.

## Acceptance Criteria

- [x] Partial dir name embeds both mode and step.
- [x] Partial dir is byte-identical-shape to final dir.
- [x] `save_partial(async_mode=True, partial=True)` returns the target
  `Path` and dispatches disk-write to a single-flight background thread.
- [x] `save_partial(async_mode=True, partial=False)` raises `ValueError`
  to enforce the final-save sync contract.
- [x] Calling `save_partial(async_mode=True)` with a prior pending Future
  auto-joins the prior Future before submitting; if the prior background
  write raised, the new call re-raises that exception.
- [x] `save_partial_join()` joins any pending Future, re-raises exceptions,
  is a no-op if no Future is pending.
- [x] CPU state_dict snapshot is deep-copied (`.detach().cpu().clone()`)
  before submission so optimizer.step() during the background write
  cannot corrupt the snapshot.
- [x] Sentinel-last invariant from LLR-0029 holds in both sync and async
  paths — the on-disk byte layout is identical.
