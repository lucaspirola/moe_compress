## Description

`kdr.quant.factory.partition_and_dispatch(model, quant_block, *,
calibration_batches, ptq_subset_size, fp32_carve_outs,
attention_module_paths, kv_quant_exempt_indices) -> list[QuantBackend]`
walks the `QuantBlock`'s weight spec (either `UniformWeightSpec` or
`MixedWeightSpec`'s `spec_map`) and produces one `QuantBlockSubset` per
backend, with the weight slot containing exactly the
`WeightPatternSpec` entries that route to that backend (in spec_map
declaration order â€” preserved via insertion-ordered grouping).

Supporting machinery (all in `factory.py`):

- `BackendName = Literal["modelopt", "native"]` â€” typo-safety for
  routing dict keys. A typo like `"modelpot"` is a mypy error, not a
  silent `dict.get()` miss.
- `WeightRoutes(by_backend: dict[BackendName, list[WeightPatternSpec]])`
  and `QuantRoutes(weight, key, value)` â€” frozen, slotted dataclasses
  (not Pydantic, to skip `tests/test_strict_typing.py` auto-discovery
  and because the inputs are already-validated specs).
- `_route_quantizers` consults
  `kdr.quant.modelopt_backend.feature_matrix.is_supported(target,
  format, bits)` per `WeightPatternSpec` and per KV quantizer, and
  assigns each to either `"modelopt"` or `"native"`.
- `_subset_for(backend, routes, key_spec, value_spec)` builds the
  per-backend `QuantBlockSubset` with weight populated from
  `routes.weight.by_backend[backend]` (or `None` if that backend has
  no weight patterns).
- `_describe_subset(subset)` logs one line per `WeightPatternSpec` in
  `weight[<pattern>]=<bits>b/<format>` form.

`partition_and_dispatch` constructs each routed backend with
adapter-supplied context (`fp32_carve_outs`, attention module paths,
KV-exempt indices) and the calibration-loop closure, invokes each
backend's `apply_quant` exactly once, and returns the list of backends
actually used so the caller (training loop) can dispatch the matching
`save` later.

## Rationale

Profile-J recipes mix four GGUF formats across distinct Linear-name
patterns (IQ2_XS / Q3_K / IQ4_XS / Q5_K). Routing must therefore
operate per-`WeightPatternSpec`, not per-quantizer-target. Even when all
weight patterns route to the same backend (the typical case â€” all GGUF
formats go to Native because the feature matrix has no GGUF entries),
the routing machinery needs to think in *sets of patterns per backend*
from day one so that future configs that fan out across backends
(e.g. NVFP4 weight via modelopt + GGUF KV via native) work without a
retrofit.

`BackendName` Literal + frozen dataclasses for `WeightRoutes` /
`QuantRoutes` are typo-safety measures: routing is a fan-out point
where a silent wrong-backend dispatch would only surface at training
time.

Insertion-ordered grouping preserves spec_map declaration order so the
native backend's "first matching pattern wins on a given Linear module"
semantics are predictable.

## Acceptance Criteria

- [ ] `BackendName = Literal["modelopt", "native"]` is the type of
      every routing-dict key in this module
- [ ] `WeightRoutes` and `QuantRoutes` are
      `@dataclass(frozen=True, slots=True)` (not Pydantic)
- [ ] `partition_and_dispatch` accepts a `QuantBlock` whose `weight` is
      either `UniformWeightSpec` or `MixedWeightSpec` and produces one
      `QuantBlockSubset` per backend that has at least one of
      weight/key/value populated
- [ ] When `MixedWeightSpec.spec_map` contains patterns whose
      `(target, format, bits)` tuples route to both backends, the
      resulting subsets carry the per-backend slice in spec_map order
- [ ] When all weight patterns route to the same backend AND that
      backend is also the routing target for K and/or V, the factory
      merges them into a single `QuantBlockSubset` (LLR-0013's
      "one subset per backend" invariant still holds)
- [ ] `_describe_subset` logs one line per `WeightPatternSpec` in
      `weight[<pattern>]=<bits>b/<format>` form
- [ ] No `isinstance(... , MixedWeightSpec) raise NotImplementedError`
      stubs remain in `factory.py`
- [ ] Test exercises a Profile-J YAML (four GGUF formats) and asserts
      all patterns route to native; modelopt subset is `None`
- [ ] Test also exercises a mixed-backend YAML (NVFP4 weight via
      modelopt + INT2 V via native) to confirm fan-out still works
