## Description

`kdr.training.zero3_init` provides a module-level dict `_DSCHF_HOLDER` keyed by `id(ds_config)` that pins `transformers.deepspeed.HfDeepSpeedConfig` instances for the lifetime of model construction, plus an `activate_zero3_init(accelerator) -> ContextManager[None]` context manager that wraps a model load such that the load happens under an active `HfDeepSpeedConfig`. Direct port from `structural_recovery/distillation.py:241–276`.

## Rationale

ZeRO-3 sharding requires that an `HfDeepSpeedConfig` instance is alive at the moment `from_pretrained` runs — otherwise the model loads as a single full-precision copy on rank 0 and the subsequent shard call no-ops silently. The `_DSCHF_HOLDER` pattern keeps a strong reference to the config object so that Python's garbage collector cannot drop it before model construction completes. The plan flags this hook ordering as **(P0)** in Phase 4: get this wrong and ZeRO-3 silently pretends to shard but doesn't, eating CPU/GPU memory until the run OOMs hours later.

This LLR is closely related to LLR-0040 (`is_zero3` detection) — `is_zero3` is the predicate; `activate_zero3_init` is the action gated on the predicate.

## Acceptance Criteria

- [ ] `_DSCHF_HOLDER` is a module-level dict keyed by `id(ds_config)`. The value-type annotation is `dict[int, object]` rather than `dict[int, HfDeepSpeedConfig]`: `transformers.deepspeed` is in the mypy `ignore_missing_imports` override list (LLR-0044) because it lacks published typed stubs, so naming `HfDeepSpeedConfig` directly in the annotation produces an `Any`-leak through the strict-typing wall. The stored values ARE `HfDeepSpeedConfig` instances at runtime; the loose annotation is documented in the source comment
- [ ] `activate_zero3_init(accelerator)` returns a `contextlib.contextmanager`-decorated function that constructs an `HfDeepSpeedConfig` from the accelerator's DeepSpeed config and stores it in `_DSCHF_HOLDER` for the duration of the `with` block
- [ ] The context is a no-op when `is_zero3(accelerator)` returns False (so single-GPU bnb-8bit runs are unaffected)
- [ ] **Required call order under ZeRO-3** (verified by a unit test that mocks `from_pretrained`, `mtq.quantize`, and `accelerator.prepare()` and records the call sequence):
  1. Enter `activate_zero3_init(accelerator)` context
  2. Inside context: `from_pretrained(teacher)`
  3. Inside context: `from_pretrained(student)`
  4. **Inside context** (still): `mtq.quantize(student, modelopt_cfg, calibrate_loop)` — `mtq.quantize` rewrites the student's module structure (wraps `Linear`/`Conv` in fake-quant modules); the rewrite must happen while `HfDeepSpeedConfig` is still active so the new modules participate in ZeRO-3 sharding rather than landing as dense replicas on rank 0
  5. Exit `activate_zero3_init` context
  6. Outside context: `accelerator.prepare(student, optimizer, scheduler, dataloader)` — DeepSpeed installs runtime hooks on the now-quantized, sharded student
- [ ] Test verifies that loading a model OUTSIDE the context (when `is_zero3` is True) produces a non-sharded model on rank 0; loading INSIDE the context produces a sharded model with `param.ds_status == ZeroParamStatus.NOT_AVAILABLE` on non-owning ranks
- [ ] The context manager handles exceptions cleanly: an exception during model load releases the `HfDeepSpeedConfig` strong reference so a subsequent retry doesn't leak memory
