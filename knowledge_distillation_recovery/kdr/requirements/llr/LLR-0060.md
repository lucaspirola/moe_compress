## Description

Two-file fix:

### 1. `kdr/src/kdr/training/loop.py`

Between `adapter.load_teacher_and_student` (Stage 1, line ~96) and
`partition_and_dispatch` (Stage 1, line ~124), add:

```python
# Under ZeRO-3, activate_zero3_init has already partitioned the model
# across ranks; an explicit .to(device) would un-partition it. Skip.
# Under DDP / single-GPU non-DS, accelerator.prepare (Stage 2) is
# normally where the device move happens, but mtq.quantize's
# calibration forward loop runs synchronously inside
# partition_and_dispatch — if the student is still on CPU at that
# point, the entire calibration runs on CPU. Pre-emptively move here
# so mtq.quantize's calibrate_forward_loop sees a GPU model. HLR-0018.
if not is_deepspeed(accelerator):
    student = student.to(accelerator.device)
```

This sits INSIDE the `activate_zero3_init` context (the `with` block
that spans Stage 1) and INSIDE the `if config.mode == "da_qad"`
branch, immediately before `partition_and_dispatch` is invoked.
HLR-0018 scopes the mandate to `da_qad` runs only (mtq.quantize
calibration is the trigger); BF16 mode does not require the early
device move and continues to rely on `accelerator.prepare` at
Stage 2 for placement.

### 2. `kdr/src/kdr/quant/factory.py`

In `_make_calibrate_loop`, wrap each batch in a `.to(device)` call
before invoking the model:

```python
def calibrate_loop(model: nn.Module) -> None:
    was_training = model.training
    model.eval()
    # Resolve the device once — under DS3 we'd use the partitioned
    # parameter's device; under non-DS the model has uniform device.
    device = next(model.parameters()).device
    try:
        with torch.no_grad():
            for batch in subset:
                model(input_ids=batch.to(device))
    finally:
        if was_training:
            model.train()
```

`next(model.parameters()).device` is the correct device resolver
under all distributed types: under ZeRO-3 the rank-local parameter
is on the current GPU; under DDP the wrapped model's first parameter
is on the local device; under no distribution the model has a
uniform device.

### What does NOT change

- `partition_and_dispatch`'s call signature
- The Stage-2 `accelerator.prepare(student, optim)`
- The Stage-3 `teacher.to(accelerator.device)` (the teacher's
  placement remains a Stage-3 concern, untouched)
- LLR-0048's ordering (apply_quant inside `activate_zero3_init`,
  before `accelerator.prepare`) — the fix sits between the load and
  the dispatch, inside that context, so the LLR-0048 invariant holds

## Rationale

The minimal-blast-radius split: the call site (training/loop.py)
owns "where the model is when factory sees it", and the factory
helper owns "where the inputs are when the model sees them". Both
are local fixes, each guarded by a single device-resolution call.

`next(model.parameters()).device` is preferred over capturing
`accelerator.device` in the closure: under DS3 the closure may be
invoked from contexts where the model's effective device differs
from the accelerator's nominal device (e.g., during the
HfDeepSpeedConfig-driven partitioning step). Reading the device
from the model itself is the canonical idiom.
