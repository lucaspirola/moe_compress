## Description

`kdr.adapters.zaya1_8b.Zaya1Adapter.fp32_carve_outs(model)` returns:

```python
return ["lm_head", "embed_tokens", "router", "norm", "val_proj"]
```

Diff vs. the v0 list (`["lm_head", "embed_tokens", "router", "norm",
"cca_cache"]`):

- `cca_cache` removed — not a module in ZAYA1; runtime tensor buffer
  only.
- `val_proj` added — matches `self_attn.qkv.val_proj1` and
  `self_attn.qkv.val_proj2` (the CCA value projections).
- `lm_head`, `embed_tokens`, `router`, `norm` retained.

The hypothesised `cca_conv` from earlier Phase 7.2 design is dropped:
the CPU dry-run of `Zyphra/ZAYA1-8B` returned zero matches for
`cca_conv`. The CCA value path is `val_proj1`/`val_proj2`, and the
`val_proj` substring is the right pattern.

The docstring is updated to enumerate `val_proj` and drop the
`cca_cache` entry. The `# REQ: LLR-0057` marker stays.

## Rationale

Substring patterns are matched against `model.named_modules()`. A
substring that matches no module is dead weight: the carve-out list
contract is "every entry resolves to at least one Linear on the
deployment model". `cca_cache` failed this for ZAYA1 (runtime buffer,
not module); `cca_conv` failed because the CCA implementation in the
Zyphra fork uses `val_proj1`/`val_proj2` instead.

`val_proj` is the smallest substring that catches both CCA value
Linears without colliding with any other module name (verified by
the CPU dry-run: no `val_proj` substring appears outside the CCA
path).

## Acceptance Criteria

- [ ] `Zaya1Adapter().fp32_carve_outs(any_model)` returns
      `["lm_head", "embed_tokens", "router", "norm", "val_proj"]`
      (or an order-independent equivalent set)
- [ ] The returned list has exactly 5 entries (no duplicates,
      `cca_conv` and `cca_cache` are absent)
- [ ] The docstring lists `val_proj` and explains its CCA-value
      target; the `cca_cache` mention is removed
- [ ] A test asserts the set equality
      `set(out) == {"lm_head", "embed_tokens", "router", "norm",
      "val_proj"}` on a `Zaya1Adapter()` instance
- [ ] A test asserts that on a ZAYA1-shaped fake model with modules
      named `self_attn.qkv.val_proj1` and `self_attn.qkv.val_proj2`,
      both submodule names contain the `val_proj` substring
- [ ] A test asserts that for EACH of the 5 carve-out substrings,
      at least one module name in a ZAYA1-shaped fake model contains
      that substring (per HLR-0015 AC: every entry resolves on the
      deployment-shape model)
- [ ] A test asserts that on a ZAYA1-shaped fake model containing
      the canonical ZAYA1 module set, the substring `val_proj` does
      NOT match any module name outside `self_attn.qkv.val_proj{1,2}`
      (no collision with non-CCA Linears)
- [ ] The previously-existing carve-out tests
      (`test_fp32_carve_outs_includes_lm_head` and
      `test_fp32_carve_outs_resolve_to_non_empty_submodule_sets`)
      continue to pass after the list update — the v0 entries
      (`lm_head`, `embed_tokens`, `router`, `norm`) are preserved
- [ ] The `# REQ: LLR-0057` marker is present on
      `fp32_carve_outs`
