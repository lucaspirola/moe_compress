## Description

`kdr.quant.modelopt_backend.ModelOptBackend` implements `QuantBackend.apply_quant(model, quant_block)` by translating its assigned `QuantBlockSubset` into a single modelopt config dict (via `config_map.py`) and calling `mtq.quantize` exactly once. It implements `save(model, output_dir)` by delegating to the converter selected by `feature_matrix` for the actual format used (see LLR-0021).

## Rationale

Default backend; the path that exists today and is tested by NVIDIA. Single `mtq.quantize` call per `apply_quant` invocation upholds HLR-0002 AC #3.

## Acceptance Criteria

- [ ] Calls `mtq.quantize(model, modelopt_cfg, calibrate_loop)` exactly once per `apply_quant` invocation (verified by a unit test that wraps `mtq.quantize` and asserts call count == 1)
- [ ] When `feature_matrix` routes BOTH the weight quantizer AND at least one of the K/V quantizers to ModelOptBackend, `factory.py` merges them into a single `QuantBlockSubset` (with multiple non-None fields) BEFORE dispatching, so ModelOptBackend's `apply_quant` is invoked exactly once with a combined sub-block and produces a single merged modelopt config dict that covers all routed quantizers. This is the scenario per LLR-0013 AC #4: the factory's partition-or-merge logic guarantees ModelOpt sees at most one combined block per run, NOT two separate calls
- [ ] `save` delegates the actual converter selection to LLR-0021's logic; this LLR makes no claim about which converter class is used for which format
