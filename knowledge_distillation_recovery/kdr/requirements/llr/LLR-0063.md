## Description

Replace the two Python loops in
`kdr/src/kdr/quant/native_backend/gguf_codebooks.py` with vectorised
PyTorch operations:

### `get_iq2xs_grid`

Before:
```python
flat: list[int] = []
for val in IQ2XS_GRID_RAW:
    for i in range(8):
        flat.append((val >> (i * 8)) & 0xff)
tensor = torch.tensor(flat, dtype=dtype, device=device).view(512, 8)
```

After:
```python
raw = torch.tensor(IQ2XS_GRID_RAW, dtype=torch.int64, device=device)
shifts = torch.arange(8, device=device, dtype=torch.int64) * 8
# (512, 8) bytes via right-shift + mask
unpacked = (raw.unsqueeze(-1) >> shifts) & 0xFF
tensor = unpacked.to(dtype)
```

### `get_ksigns_iq2xs`

Before:
```python
expanded: list[list[float]] = []
for bits in KSIGNS_IQ2XS:
    row = [-1.0 if (bits >> i) & 1 else 1.0 for i in range(8)]
    expanded.append(row)
tensor = torch.tensor(expanded, dtype=dtype, device=device)
```

After:
```python
raw = torch.tensor(KSIGNS_IQ2XS, dtype=torch.int64, device=device)
shifts = torch.arange(8, device=device, dtype=torch.int64)
# bit_set[i, j] = 1 if bit j of byte i is set else 0
bit_set = ((raw.unsqueeze(-1) >> shifts) & 1)
# Map 1 → -1.0 and 0 → +1.0 (per ggml convention)
tensor = (1 - 2 * bit_set).to(dtype)
```

The cache lookup + early-return remain unchanged. Both
implementations preserve the existing per-(device, dtype) cache
keys.

## Rationale

Bit-extraction and byte-extraction are integer arithmetic; in both
the Python-loop and tensor-op forms the integer values produced are
identical. Casting `.to(dtype)` at the end converts to the requested
float dtype, matching the prior `torch.tensor(list, dtype=dtype)`
behaviour.

The vectorised form expresses the byte/bit-unpack via a single
broadcast over the `shifts` arange, dispatching one or two CUDA
kernels per cache-miss instead of materialising a Python list and
then doing an H2D copy.

## Acceptance Criteria

- [ ] `get_iq2xs_grid`'s body no longer contains
      `for val in IQ2XS_GRID_RAW:` or any nested `for i in range(8):`
      loop; the unpack uses `torch.arange` + bitwise tensor ops
- [ ] `get_ksigns_iq2xs`'s body no longer contains
      `for bits in KSIGNS_IQ2XS:` or any list-comp over a
      `range(8)`; the unpack uses `torch.arange` + bitwise tensor ops
- [ ] Returned tensor shapes are unchanged: `(512, 8)` for the grid,
      `(128, 8)` for ksigns
- [ ] Returned tensor dtypes match the `dtype` kwarg
- [ ] A pytest test in `tests/property/test_iq2xs_tile_equivalence.py`
      (or a new `tests/test_gguf_codebooks_unpack.py`) asserts each
      new implementation is `torch.equal` to an inline reference
      that replays the old Python-loop algorithm, for both
      `(torch.device("cpu"), torch.float32)` and
      `(torch.device("meta"), torch.float32)` pairs (the meta-device
      pair exercises the cache+dispatch path without requiring real
      hardware)
- [ ] The per-(device, dtype) cache contract is preserved: a second
      call with the same `(device, dtype)` returns the SAME tensor
      object (verified by an `is` identity test)
- [ ] Existing tests in `tests/property/test_ste_simulators.py` and
      `tests/property/test_iq2xs_tile_equivalence.py` continue to
      pass (they consume these codebook tensors transitively via
      the IQ2_XS STE)
- [ ] No new `# type: ignore` comments anywhere — including the
      implementation file (`kdr/src/kdr/quant/native_backend/gguf_codebooks.py`),
      any new helper module touched by this LLR, and any new test
      file added
