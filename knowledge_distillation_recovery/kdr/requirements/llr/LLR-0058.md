## Description

`kdr/configs/zaya1_8b_da_qad_profileJ_gguf.yaml` is the Profile-J
recovery recipe for ZAYA1-8B. Its diff against
`zaya1_8b_da_qad_nvfp4_int4kv.yaml` is restricted to `quant.weight`:

```yaml
quant:
  weight:
    spec_map:
      - pattern: linear_fc1   # MoE expert gate+up (fused)
        bits: 2
        format: iq2_xs
        granularity: block
        transform: none
      - pattern: linear_fc2   # MoE expert down
        bits: 3
        format: q3_k
        granularity: block
        transform: none
      - pattern: linear_q     # attention query
        bits: 4
        format: iq4_xs
        granularity: block
        transform: none
      - pattern: linear_k     # attention key
        bits: 4
        format: iq4_xs
        granularity: block
        transform: none
      - pattern: o_proj       # attention output
        bits: 4
        format: iq4_xs
        granularity: block
        transform: none
      - pattern: embed_tokens # tied with lm_head
        bits: 5
        format: q5_k
        granularity: block
        transform: none
      - pattern: lm_head      # tied with embed_tokens
        bits: 5
        format: q5_k
        granularity: block
        transform: none
  kv_quant:
    key:
      bits: 4
      format: int
      granularity: channel
      transform: none
    value:
      bits: 4
      format: int
      granularity: token
      transform: none
```

The non-quant blocks (`teacher`, `student`, `calibration`,
`distillation`, `eval`) are copied verbatim from the NVFP4 reference,
**except** `teacher.name_or_path` and `student.source` which target
`Zyphra/ZAYA1-8B` (uniform per-layer architecture). The
`reasoning-base` variant used per-layer interleaved config lists
(`cca_num_q_heads: [8, 0, 8, …]`, `num_hidden_layers: 120`) that the
Zyphra transformers fork does not support — verified live via shape
mismatch crash at teacher weight load on instance 36666031.

The leading comment block documents (a) Profile-J intent, (b) the
ZAYA1 module-name mapping from the canonical-HF design table, and
(c) the runtime-KV-quant `-ctk q4_0 -ctv q4_0` note for llama.cpp.

The YAML must validate via the existing `Config.model_validate`
path — no schema changes required (LLR-0010 already accepts
`MixedWeightSpec`).

## Rationale

The pattern set was chosen by CPU dry-run of `Zyphra/ZAYA1-8B`:
- `linear_fc1`/`linear_fc2` are unique substrings appearing only in
  the MoE expert path (16 experts × 80 layers = 1280 modules each).
- `linear_q` / `linear_k` are unique substrings appearing only at
  `self_attn.qkv.linear_q` / `self_attn.qkv.linear_k` (80 layers
  each).
- `o_proj` matches only `self_attn.o_proj` (80 layers).
- `embed_tokens` and `lm_head` cover the tied input/output embedding
  pair.

The substring uniqueness was checked: `linear_q` is not a substring
of `linear_fc1` (or any other module name), and `linear_k` is not a
substring of any other name. So first-match-wins precedence is
inconsequential here — every pattern matches a disjoint module set.

CCA value projections (`val_proj1`/`val_proj2`), the routing
softmax module tree (`router`/`router.*`), and norms are intentionally
absent from the `spec_map`; they are handled by the ZAYA1 adapter's
`fp32_carve_outs` list (LLR-0057): "val_proj", "router", "norm".

## Acceptance Criteria

- [ ] The file `kdr/configs/zaya1_8b_da_qad_profileJ_gguf.yaml`
      exists at that path
- [ ] A pytest test loads the file via
      `yaml.safe_load(open(<path>))` → `Config.model_validate(...)`
      and asserts no `ValidationError`
- [ ] The same test asserts `config.mode == "da_qad"`
- [ ] The same test asserts
      `isinstance(config.quant.weight, MixedWeightSpec)`
- [ ] The same test asserts `len(config.quant.weight.spec_map) == 7`
- [ ] The same test asserts the spec_map entries' `.pattern` values
      equal the set
      `{"linear_fc1", "linear_fc2", "linear_q", "linear_k",
        "o_proj", "embed_tokens", "lm_head"}`
- [ ] The same test asserts each pattern's
      `(bits, format, granularity)` triple matches:
      - `linear_fc1` → `(2, "iq2_xs", "block")`
      - `linear_fc2` → `(3, "q3_k", "block")`
      - `linear_q`/`linear_k`/`o_proj` → `(4, "iq4_xs", "block")`
      - `embed_tokens`/`lm_head` → `(5, "q5_k", "block")`
- [ ] The same test asserts the KV block:
      K `(bits=4, format='int', granularity='channel', transform='none')`;
      V `(bits=4, format='int', granularity='token', transform='none')`
- [ ] The same test asserts
      `config.student.source == "Zyphra/ZAYA1-8B"`
- [ ] No `spec_map.*.pattern` contains `val_proj`, `router`, or
      `norm` (those are carve-outs, LLR-0057)
- [ ] Substring-disjointness assertion in the test:
      `"linear_q" not in "linear_fc1"` and vice-versa for every
      pair of spec_map patterns (i.e., no spec_map pattern is a
      substring of another, so first-match-wins is inconsequential)
