## Description

### `_build_quantization_config` (`kdr.io.save`)

The Phase-7.2 stopgap branch
`if isinstance(quant_block.weight, MixedWeightSpec): raise NotImplementedError(...)`
is **deleted**. The function dispatches on the union:

- `UniformWeightSpec` branch: unchanged from v0. Emits one
  `config_groups.group_0` with `targets: ["Linear"]` and
  `weights = _weight_spec_to_ct(quant_block.weight)`.
- `MixedWeightSpec` branch: walks `quant_block.weight.spec_map` and
  groups entries by `(format, bits, granularity)` triple using an
  insertion-ordered `dict[tuple, list[str]]` keyed on the triple and
  valued by the ordered list of patterns. The first time a triple is
  seen, a new group is appended; subsequent same-triple entries append
  their pattern to that group's `targets` list. Each group becomes
  one `config_groups.group_N` entry where N is the insertion index
  (0-based). `weights` is built from the first `WeightPatternSpec`
  whose triple matched — since the triple defines the recipe, any spec
  in the group serializes identically through `_weight_spec_to_ct`.

`_weight_spec_to_ct` is widened to accept
`UniformWeightSpec | WeightPatternSpec` (both expose `bits`, `format`,
`granularity`); existing call sites are unchanged.

Defense-in-depth: if `quant_block.weight` is neither branch of the
union (impossible at runtime given Pydantic validation, but type
checkers can't always prove it), the function raises
`ValueError(f"unexpected weight type {type(...).__name__}")`.

### `NativeBackend.save` (`kdr.quant.native_backend.backend`)

The Phase-4 stopgap `raise NotImplementedError(...)` is **deleted** for
the common case (weight subset present). The new body:

```
def save(self, model: nn.Module, output_dir: Path) -> None:
    unwrapped = model  # Accelerator unwrap is the caller's concern.
    state_dict = {
        name: tensor.detach().to(torch.bfloat16)
        for name, tensor in unwrapped.state_dict().items()
    }
    unwrapped.save_pretrained(
        output_dir, state_dict=state_dict, safe_serialization=True
    )
```

Reading `unwrapped.state_dict()` while the parametrizations from
LLR-0055 are active returns the fake-quanted weight tensors directly
(PyTorch's `parametrize` machinery routes `module.weight` through the
parametrization's forward on every access, and `state_dict()` reads
`module.weight`). Casting to BF16 ensures storage is the platform's
inference dtype; the simulated-quant snap is already baked in.

### Import changes in `save.py`

The existing `from ..quant.specs import KVQuantSpec, MixedWeightSpec,
WeightQuantSpec` becomes
`from ..quant.specs import KVQuantSpec, MixedWeightSpec,
UniformWeightSpec, WeightPatternSpec, WeightQuantSpec`. The
`MixedWeightSpec` import is still needed (the function branches on
its type).

## Rationale

Grouping by `(format, bits, granularity)` triple matches
compressed-tensors' own grouping semantics and produces a compact
`quantization_config` block whose group count tracks distinct recipes,
not pattern count. The insertion-ordered dict preserves spec_map
declaration order across groups, so a config consumer reading
`group_0`, `group_1`, …, sees the same priority order the user wrote.

`weights` taking the first matching spec is correct because group
membership is defined by recipe-equivalence — any spec in the group
yields the same `_weight_spec_to_ct` output, so picking the first is
deterministic.

`NativeBackend.save`'s BF16 cast is intentional: the simulated-quant
result is generally a small magnitude shift of the original BF16/FP32
weight, so storing in BF16 is lossless for the inference target while
keeping file size at the model's nominal byte count (Profile-J's
~3.25 GB on-disk estimate assumes BF16 storage for the simulated
artifact prior to GGUF conversion). The downstream GGUF converter
(Task 9 / HLR-0017) reads back the BF16 tensors and re-encodes them
per `quantization_config`.

## Acceptance Criteria

- [ ] The `isinstance(quant_block.weight, MixedWeightSpec)` guard in
      `_build_quantization_config` is removed; both union branches are
      handled
- [ ] For a `MixedWeightSpec` with four distinct
      `(format, bits, granularity)` triples, `config_groups` has
      exactly four entries named `group_0`..`group_3`
- [ ] For a `MixedWeightSpec` whose entries share triples,
      same-triple patterns collapse to one group with a multi-element
      `targets` list preserving declaration order
- [ ] The Uniform path's output is functionally equivalent to v0 (the
      emitted `config_groups.group_0` dict equals the v0 dict
      structurally, ignoring JSON key-ordering and whitespace
      differences)
- [ ] `_weight_spec_to_ct` accepts both `UniformWeightSpec` and
      `WeightPatternSpec`; no two parallel implementations
- [ ] `NativeBackend.save` no longer raises `NotImplementedError`
      when invoked on a model with a weight subset
- [ ] `NativeBackend.save` calls `unwrapped.save_pretrained(output_dir,
      state_dict=..., safe_serialization=True)` with a state_dict
      whose tensors are BF16
- [ ] A pytest test stages a model with a real registered
      parametrization, calls `NativeBackend.save`, and asserts that
      the resulting safetensors directory contains a `model.safetensors`
      (or shard) file
- [ ] A pytest test on `_build_quantization_config` confirms the
      Profile-J four-group emission with the expected `targets` lists
- [ ] A pytest test on `_build_quantization_config` confirms the
      same-triple-collapse case (two entries with identical triples,
      distinct patterns → one group with two targets)
- [ ] Calling `_build_quantization_config` with a `quant_block.weight`
      whose runtime type is neither `UniformWeightSpec` nor
      `MixedWeightSpec` raises `ValueError` containing the offending
      type name (defense-in-depth against type-checker holes; Pydantic
      validation prevents the case in practice)
- [ ] The emitted `quantization_config` dict's `ignore` key is
      `list(fp32_carve_outs)` and `kv_cache_scheme` is
      `{'key': _kv_spec_to_ct(...), 'value': _kv_spec_to_ct(...)}` —
      both shapes unchanged from the v0 Uniform path
- [ ] An end-to-end pytest test stages a stand-in model with a 2-entry
      `MixedWeightSpec`, calls `save_kdr_artifact` (which invokes
      `NativeBackend.save` + `_inject_quantization_config`), and
      asserts both (a) the safetensors file exists and (b) the
      resulting `config.json::quantization_config.config_groups` has
      two entries with the expected `targets` lists

## Notes

Non-normative; not tested:

If the dispatched subset carries NO weight (Native used only for KV
hooks), `save` may no-op and return silently — the caller's
`_inject_quantization_config` still runs to record the KV scheme into
`config.json`.
