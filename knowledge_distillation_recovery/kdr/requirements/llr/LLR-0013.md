## Description

`kdr.quant.interface.QuantBackend` is a `typing.Protocol` declaring
`apply_quant(model, quant_block: QuantBlockSubset) -> None`,
`save(model, output_dir: Path) -> None`, and `name: str`.

`QuantBlockSubset` is a Pydantic v2 strict model
(`ConfigDict(strict=True, extra='forbid')`) with three **flat optional
fields**:

- `weight: list[WeightPatternSpec] | None` — the per-backend slice of the
  weight spec_map. A `_no_empty_list` field validator rejects
  `weight = []` (empty list is semantically distinct from `None` and
  would break `is_empty()`). The Uniform-weight case (existing YAMLs
  with a single global weight spec) normalises to a single-entry list
  with `pattern=""` (the catch-all pattern that matches every Linear);
  the Mixed case carries the routed subset of the original `spec_map`.
- `key: KVQuantSpec | None`
- `value: KVQuantSpec | None`

The flat shape (rather than nested `kv_quant.key`/`kv_quant.value`) lets
`factory.py` build per-quantizer sub-blocks without constructing
intermediate `kv_quant` containers. The backend acts only on populated
(non-None) fields. `is_empty()` checks
`weight is None and key is None and value is None`.

The factory shim `factory._normalize_weight_to_patterns(weight)` is the
load-bearing piece of backward compatibility: it accepts either branch
of the `UniformWeightSpec | MixedWeightSpec` union (LLR-0010) and
returns the canonical `list[WeightPatternSpec]` form. Uniform configs
collapse to a one-entry list with `pattern=""`; Mixed configs pass
their `spec_map` through unchanged.

## Rationale

Single `apply_quant` method (rather than `apply_weight_quant` +
`apply_kv_quant`) preserves the "exactly one `mtq.quantize` call"
invariant from HLR-0002 AC #3 without sacrificing the ability to mix
backends per quantizer — the factory partitions the YAML's quant_block
and dispatches a flat sub-block to each backend. Decouples the training
loop from any specific quantization library.

The flat field shape is intentional and asymmetric to the YAML's nested
`quant.kv_quant.{key,value}` structure: the YAML mirrors how humans
think about the recipe (K and V belong together under "kv"), while the
runtime sub-block flattens them so the backend sees three peers and can
act on each independently.

The `weight` slot is a *list* (not a single spec) because Profile-J
recipes require per-module-pattern quantization (IQ2_XS for
`gate_proj/up_proj`, Q3_K for `down_proj`, etc.). Per-pattern routing
(LLR-0017) needs to hand each backend a list of the patterns that
routed to it; representing that as a list everywhere — with the uniform
case collapsing to one entry — removes the type bifurcation between
uniform and mixed paths.

## Acceptance Criteria

- [ ] Protocol declares exactly two methods (`apply_quant`, `save`)
      plus a `name: str` attribute
- [ ] `QuantBlockSubset` is a Pydantic v2 strict model
      (`ConfigDict(strict=True, extra='forbid')`) with EXACTLY three
      flat optional fields named `weight`, `key`, `value` (no nested
      `kv_quant` container)
- [ ] `weight: list[WeightPatternSpec] | None`; the `_no_empty_list`
      field validator raises `ValidationError` when `weight = []`
- [ ] `QuantBlockSubset(weight=None, key=None, value=None).is_empty()`
      returns True
- [ ] A backend that receives a `QuantBlockSubset` with all three
      fields == None raises `ValueError("backend received an empty
      quant block")`
- [ ] `factory._normalize_weight_to_patterns(UniformWeightSpec(...))`
      returns a single-entry `list[WeightPatternSpec]` with
      `pattern == ""` and the other four fields copied verbatim
- [ ] `factory._normalize_weight_to_patterns(MixedWeightSpec(spec_map=[s1, s2]))`
      returns `[s1, s2]` (order preserved)
- [ ] `factory.py`'s partition-or-merge logic produces ONE
      `QuantBlockSubset` per backend, not per quantizer. When
      `feature_matrix` routes the three quantizers to two backends
      (e.g. ModelOpt for weight, Native for K and V), the factory emits
      `QuantBlockSubset(weight=[...], key=None, value=None)` to
      ModelOpt and `QuantBlockSubset(weight=None, key=..., value=...)`
      to Native — never two separate dispatches to the same backend.
      When all three quantizers route to ModelOpt the factory emits a
      single `QuantBlockSubset(weight=[...], key=..., value=...)` so
      ModelOptBackend's `apply_quant` is still invoked exactly once
- [ ] Existing uniform-weight YAML configs continue to load and reach
      `apply_quant` without per-pattern logic firing (the catch-all
      `pattern=""` matches every Linear)
- [ ] mypy --strict accepts both `ModelOptBackend` and `NativeBackend`
      as conforming
