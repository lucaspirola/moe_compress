## Description

Single-file change to
`kdr/src/kdr/quant/native_backend/ste_simulators.py`:

1. Replace the `_IQ2XS_ARGMIN_TILE: int = 64` constant with
   `_IQ2XS_ARGMIN_TILE_MAX: int = 8192`. Add a comment recording
   the per-tile memory budget arithmetic:
   ```
   # Each tile contributes two (tile_n, 65536) fp32 intermediates
   # (`inner` and `dist`) → tile_n × 65536 × 4 bytes × 2 ≈ tile_n
   # × 512 KB peak per tile. With max = 8192 the per-tile peak is
   # ~4 GB, well within the headroom on an H200/B200 after teacher
   # and student weights are resident (~140 GB free of 141-180 GB).
   ```

2. In `_iq2xs_snap_block`, replace the existing loop body's use of
   `_IQ2XS_ARGMIN_TILE` with `_IQ2XS_ARGMIN_TILE_MAX`. The loop
   body is otherwise unchanged. For inputs with `n_total ≤ 8192`,
   the loop naturally executes a single iteration.

No other call sites or downstream code touch the constant; the
rename is internal-only and limited to this single function.

## Rationale

The Python-loop overhead amortisation does not change the function's
numerical output: the tiling is purely a memory-bounded chunking of
the same matmul+argmin. Increasing the tile from 64 to 8192 reduces
the number of iterations by 128× without changing arithmetic.

`_IQ2XS_ARGMIN_TILE_MAX` is named with the `_MAX` suffix so future
readers see that smaller tiles are a possibility (e.g., for
constrained memory environments), without forcing a per-call
device-memory probe.

The 8192 value is chosen as the largest power of 2 such that the
per-tile `inner` allocation (tile × 65536 × 4 bytes = 2 GB) plus
the `dist` allocation (another 2 GB) totals 4 GB peak — comfortably
within the residual VRAM on a fully-loaded teacher+student run on
141 GB (H200) and 180 GB (B200) GPUs.

## Acceptance Criteria

- [ ] `_IQ2XS_ARGMIN_TILE` is renamed to `_IQ2XS_ARGMIN_TILE_MAX`
      (no remaining reference to the old name in the file)
- [ ] The constant's value is `8192`
- [ ] A multi-line comment near the constant explains the per-tile
      memory budget: tile × 65536 × 4 bytes × 2 intermediates
      ≈ tile × 512 KB peak per tile; max=8192 → ~4 GB peak per
      tile; rationale ties this to the H200/B200 residual-VRAM
      after teacher+student load
- [ ] The loop body in `_iq2xs_snap_block` references the new
      constant name; no other code path is modified
- [ ] A pytest test asserts `_iq2xs_snap_block` output is
      `torch.equal`-bit-identical for a curated set of fixed-seed
      inputs at shapes `(1, 256)`, `(1, 512)`, `(2, 4096)`, and
      `(1, 8192)` — the reference output is captured against the
      pre-refactor implementation by re-running it inline within
      the test (using a local helper that replays the old tiled
      loop with `_IQ2XS_ARGMIN_TILE = 64`)
- [ ] A Hypothesis property test (in
      `tests/property/test_native_parametrizations.py` or a new
      `tests/property/test_iq2xs_tile_equivalence.py`) randomises
      the last-axis shape over multiples of 256 in [256, 8192] and
      the random seed, then asserts the new tiled output is
      `torch.equal` to a locally-defined reference helper that
      replays the pre-refactor algorithm with `tile = 64`
- [ ] The existing 4 hypothesis property tests in
      `tests/property/test_native_parametrizations.py` AND every
      test in `tests/property/test_ste_simulators.py` continue to
      pass without modification (they verify the parametrization-
      class wrappers, which delegate to `iq2_xs_quant_ste` →
      `_iq2xs_snap_block`)
- [ ] No new `# type: ignore` comments introduced in
      `ste_simulators.py`
