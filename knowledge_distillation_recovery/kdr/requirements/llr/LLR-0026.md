## Description

Each adapter declares a `required_attn_implementation(mode) -> 'eager' | 'sdpa'`
method that the trainer uses to pick the HF `attn_implementation` value passed
to `from_pretrained` for both teacher and student loaders. The implementation
is **mode-aware**:

- `mode == 'da_qad'` → returns the backend whose K/V tensors are exposed at a
  Python hook boundary (typically `'eager'` for models like ZAYA1 whose CCA
  layer routes through `cca_eager_attention_forward`). Required so that the
  KV-quant simulator hooks can intercept post-projection K/V tensors at every
  attention layer.
- `mode == 'bf16'` → returns the fastest backend that produces numerically
  equivalent output (typically `'sdpa'` on Hopper / Blackwell). No hooks are
  placed in pure-BF16 distillation (`NoOpReplayContextManager` only), so the
  eager-only constraint does not apply.

Flash-attn (`'flash_attention_2'`) is rejected in both modes because it fuses
the K/V projection internally and never exposes post-projection K/V tensors,
silently no-opping the KV simulator in `da_qad` and producing no measurable
speedup over SDPA on Hopper/Blackwell for the workloads we care about.

## Rationale

Phase 7.1 confirmed empirically on real ZAYA1-8B that:

1. The Zyphra fork's CCA layer uses `cca_eager_attention_forward` wired only
   under the `eager` HF backend; KV-quant hooks therefore require `eager` in
   `da_qad` mode.
2. Pure-BF16 self-distillation installs `NoOpReplayContextManager()`
   (`kdr/training/loop.py:160`) and places no hooks. Forcing eager in this
   mode unnecessarily uses the O(N²) reference attention kernel and was the
   dominant wall-time cost in the Phase 7.1 200-step variant (~33 s/step on
   H200). SDPA on Hopper/Blackwell delivers an estimated 2-3× speedup with
   numerically equivalent output (PyTorch's SDPA agrees with eager to within
   noise an order of magnitude below the BF16 forward's own non-determinism).

Making the method mode-aware preserves the original `da_qad` hookability
invariant while removing the speed tax from the BF16 path.

## Acceptance Criteria

- [x] Protocol signature in `kdr/adapters/base.py` reads
  `required_attn_implementation(mode: Mode) -> Literal['eager', 'sdpa']`.
- [x] `Zaya1Adapter.required_attn_implementation(mode='da_qad')` returns `'eager'`.
- [x] `Zaya1Adapter.required_attn_implementation(mode='bf16')` returns `'sdpa'`.
- [x] `Zaya1Adapter.load_teacher_and_student(..., mode=mode)` threads `mode` into
  the `required_attn_implementation` call AND into the
  `from_pretrained(..., attn_implementation=...)` invocation for both teacher
  and student. The trainer does NOT rely on
  `model.config.attn_implementation` being populated post-load (some HF
  versions do not store it on the config) — the assertion is a
  constructor-time invariant.
- [x] Trainer (`kdr/training/loop.py`) passes `mode=config.mode` to
  `adapter.load_teacher_and_student(...)`.
- [x] Unit tests in `kdr/tests/test_adapter_attn.py` cover the four cells of
  `{Zaya1Adapter} × {bf16, da_qad}` (just the `required_attn_implementation`
  method; the load-model assertions are GPU-only and run in Phase 7
  validation).
- [x] Flash-attn remains rejected: any `mode` value that is not `'bf16'` or
  `'da_qad'` raises rather than silently defaulting.
