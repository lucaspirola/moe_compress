## Description

`kdr.io.run_id.derive_run_id(config_path, student_repo_sha, mode) -> str` returns the first 16 hex chars of `sha256(canonical_yaml_dump(config).encode('utf-8') + b'\x00' + student_sha.encode('utf-8') + b'\x00' + mode.encode('utf-8'))`. The `\x00` byte separator prevents any field-length-collision class where two different `(yaml, sha, mode)` tuples hash to the same value because their concatenations alias.

## Rationale

Stable hash over inputs makes resume deterministic. The explicit null-byte separator is a defensive measure against the (improbable but real) class of collisions arising from variable-length YAML dumps where a longer first field could accidentally absorb the prefix of the second.

## Acceptance Criteria

- [ ] Whitespace and comment changes in the YAML do NOT change the hash (canonical Pydantic `model_dump_json` is used, not raw YAML text)
- [ ] Semantic config changes (e.g., `total_tokens` or `bits` value change) DO change the hash
- [ ] Same `(config, sha, mode)` triple → same hash on different machines (verified by a deterministic unit test with hard-coded inputs and a hard-coded expected hash)
- [ ] Field separator is the literal byte `\x00`, asserted by a unit test that constructs a tuple `(yaml="ab", sha="cde", mode="f")` and a tuple `(yaml="abc", sha="de", mode="f")` and checks that the hashes differ
