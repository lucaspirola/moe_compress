## Description

`kdr.quant.native_backend.backend.NativeBackend._install_weight_quant`
is rewritten to accept `list[WeightPatternSpec]` and walk every
`nn.Linear` in the model. The Task-3 stopgap branch in
`NativeBackend.apply_quant` that raises `NotImplementedError` on
multi-entry / non-empty-pattern lists is **deleted**.

Algorithm:

```
for spec in spec_map:
    validate (format, granularity, transform) â€” raise NotImplementedError eagerly
for name, module in model.named_modules():
    if not isinstance(module, nn.Linear): continue
    # 1. Explicit non-empty pattern wins (Profile-J priority).
    explicit = first p in spec_map where p.pattern != "" and p.pattern in name
    if explicit is not None:
        install parametrization for explicit
        continue
    # 2. Carve-out wins over the catch-all fallback (v0 back-compat).
    if self._is_carved_out(name):
        continue
    # 3. Empty-pattern fallback (uniform-config shim).
    fallback = first p in spec_map where p.pattern == ""
    if fallback is not None:
        install parametrization for fallback
        continue
    # 4. No silent FP16 fallback â€” fail loud.
    raise ValueError(f"Linear {name!r} matched no spec_map pattern and "
                     f"no fp32_carve_outs entry. Either add a pattern "
                     f"that matches or list it under fp32_carve_outs.")
```

The two helpers exposing this precedence are:

- `_first_explicit_match(name, specs) -> WeightPatternSpec | None`:
  returns the first spec with non-empty `pattern` whose substring is
  in `name`; empty-pattern entries are skipped.
- `_fallback_spec(specs) -> WeightPatternSpec | None`: returns the
  first spec with `pattern == ""`, or None.

Format-to-parametrization dispatch in `_install_weight_quant`:

| `spec.format` | parametrization class | underlying STE |
|---|---|---|
| `int` | `_IntQuantWeight(bits, axis=0)` (existing) | `int_quant_ste` |
| `iq2_xs` | `_IQ2XSQuantWeight()` | `iq2_xs_quant_ste(axis=-1)` |
| `q3_k` | `_Q3KQuantWeight()` | `q3_k_quant_ste(axis=-1)` |
| `iq4_xs` | `_IQ4XSQuantWeight()` | `iq4_xs_quant_ste(axis=-1)` |
| `q5_k` | `_Q5KQuantWeight()` | `q5_k_quant_ste(axis=-1)` |

Any other `Format` Literal value reachable here (the 17 other GGUF
formats accepted by LLR-0010 but not yet implemented) raises
`NotImplementedError(f"NativeBackend weight format={spec.format!r} "
"accepted by schema but no STE simulator ships in Phase 7.2 â€” "
"only int, iq2_xs, q3_k, iq4_xs, q5_k are wired here.")` BEFORE
attempting installation.

Granularity is interpreted per-format:

- `int` keeps the v0 contract: `granularity='channel'` only;
  other granularities raise `NotImplementedError`.
- The four GGUF formats use `granularity='block'` to signal "the format's
  own super-block-of-256 layout is the per-slice unit"; other
  granularities for a GGUF format raise `NotImplementedError`. The
  parametrizations themselves do not consume `granularity` directly
  (the simulators encode the layout) â€” it is a config-coherence check.

The four new parametrization classes are pure-pass-throughs to the
underlying STE: their `__init__` stores no per-spec state beyond what
the STE needs (in practice nothing; `axis=-1` is hard-coded for GGUF
formats per LLR-0015's axis convention). They live next to
`_IntQuantWeight` in `backend.py`.

`_is_carved_out` is unchanged (substring match against
`fp32_carve_outs`).

The per-Linear install order is the order returned by
`model.named_modules()`, which is registration order â€” stable across
runs.

## Notes

*Non-normative implementation note:* the backend **may emit** one
`log.info` summary line per call with the count of Linears installed
per format (e.g., `"NativeBackend: installed weight STE on 32 modules
(iq2_xs=16, q3_k=8, iq4_xs=8)"`), replacing the existing single-line
`INT only` log. This is observability guidance only and is not an
acceptance criterion.

## Rationale

The walk-once-iterate-spec_map structure keeps the inner loop O(NÂ·M)
in (Linears Ă— patterns) with N small (a few hundred) and M small (â‰¤ 7
in Profile J). A more elaborate trie isn't justified at this scale.

Hard-coded `axis=-1` for the GGUF parametrizations is correct because
ggml super-blocks lie on the in-features axis of an `nn.Linear` weight
(LLR-0015); exposing axis as a per-spec field would be a footgun.

The eager `NotImplementedError` for unwired GGUF formats serves as a
clear "schema accepted; backend not yet wired" signal that won't
silently degrade to FP16.

The eager `ValueError` for unmatched non-carve-out modules implements
locked design choice #5 (no silent FP16 fallback). The error message
names the module so the user can either add a pattern or extend
`fp32_carve_outs`.

## Acceptance Criteria

- [ ] `NativeBackend.apply_quant`'s multi-entry / non-empty-pattern
      `NotImplementedError` branch is removed; the function calls
      `_install_weight_quant(model, quant_block.weight)` directly
      whenever `quant_block.weight is not None`
- [ ] `_install_weight_quant`'s signature is
      `(self, model: nn.Module, specs: list[WeightPatternSpec]) -> None`
- [ ] `_install_weight_quant` raises `ValueError` with a message
      containing the unmatched module's qualified name when a non-carve-out
      Linear matches no pattern
- [ ] `_install_weight_quant` raises `NotImplementedError` when a
      `WeightPatternSpec.format` is in the GGUF set but not in
      {`iq2_xs`, `q3_k`, `iq4_xs`, `q5_k`, `int`}
- [ ] First-match-wins precedence is observable: a spec_map of
      `[(pattern="proj", iq4_xs), (pattern="q_proj", q5_k)]` installs
      IQ4_XS on a Linear named `attn.q_proj`
- [ ] Carve-out precedence is observable: a Linear matching a carve-out
      substring and no spec_map pattern installs nothing (no error)
- [ ] Carve-out wins over empty-pattern fallback: a spec_map of
      `[(pattern="", int, granularity='channel')]` with
      `fp32_carve_outs=["lm_head"]` installs `_IntQuantWeight` on a
      non-`lm_head` Linear but skips a Linear named `lm_head` (v0
      uniform-config back-compat)
- [ ] Explicit pattern wins over carve-out: a spec_map of
      `[(pattern="lm_head", q5_k, granularity='block')]` with
      `fp32_carve_outs=["lm_head"]` installs `_Q5KQuantWeight` on
      `lm_head` (locked design choice #5 â€” explicit per-pattern
      overrides the carve-out)
- [ ] Four new classes `_IQ2XSQuantWeight`, `_Q3KQuantWeight`,
      `_IQ4XSQuantWeight`, `_Q5KQuantWeight` exist in
      `native_backend/backend.py`. Each subclasses `nn.Module`, has an
      empty `__init__(self) -> None` (no constructor args beyond
      `self`), and a `forward(self, w: torch.Tensor) -> torch.Tensor`
      that delegates to its corresponding STE with `axis=-1`
- [ ] Each parametrization is installed via
      `nn.utils.parametrize.register_parametrization(module, "weight",
      cls())` â€” verified by asserting
      `nn.utils.parametrize.is_parametrized(module, "weight")` is True
      after `_install_weight_quant` returns for every Linear that
      matched (explicit or fallback)
- [ ] A pytest unit test on a stand-in module with three Linears
      `mlp.gate_proj`, `mlp.up_proj`, `mlp.down_proj` and a representative
      subset of the Profile-J spec_map (the MLP three-tensor slice:
      gate_proj=IQ2_XS, up_proj=IQ2_XS, down_proj=Q3_K)
      verifies: gate/up receive `_IQ2XSQuantWeight`, down
      receives `_Q3KQuantWeight`; each Linear has exactly one
      registered parametrization on `weight`
- [ ] A regression test that loads a v0-style uniform YAML (one weight
      spec, no spec_map) asserts that every non-carve-out Linear gets
      `_IntQuantWeight` registered on its `weight` parametrization slot
- [ ] A hypothesis property test confirms that for each of the four new
      parametrizations, `param.forward(w)` is bit-identical to the
      direct STE call with `axis=-1`
- [ ] No new `# type: ignore` comments
- [ ] `_install_weight_quant` accepts a `list[WeightPatternSpec]` of
      any length â‰Ą 1 (single-entry, multi-entry, with or without empty
      pattern) and installs without raising on the supported-format set
- [ ] On a stand-in module where two distinctly-named Linears both
      match the same `WeightPatternSpec.pattern` substring, both
      Linears receive the parametrization corresponding to that spec
- [ ] `_install_weight_quant` raises `NotImplementedError` when a
      `WeightPatternSpec.granularity` value is incompatible with its
      `format` (e.g., `int` with `granularity='block'`, or any GGUF
      format with `granularity='channel'`)
- [ ] `_install_weight_quant` raises `NotImplementedError` when a
      `WeightPatternSpec.transform` value is not `'none'` (the only
      transform value LLR-0010 admits in v0; future v1+ transforms
      remain unsupported until explicitly wired)
