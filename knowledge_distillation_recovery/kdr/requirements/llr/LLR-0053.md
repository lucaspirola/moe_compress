## Description

`kdr.quant.native_backend.gguf_codebooks` is a constants-only module that
mirrors the codebook tables used by llama.cpp's I-quant decoders. The
tables are transcribed verbatim from a pinned upstream commit of
`ggml/src/ggml-common.h`; the commit SHA is recorded in the module
docstring as the audit trail for which encoder/decoder version is in
use.

Required tables:

- `IQ2XS_GRID_RAW: tuple[int, ...]` — the 512-entry uint64 packing of the
  IQ2_XS magnitude grid (8 magnitudes per uint64 entry, ggml's compact
  layout).
- `KSIGNS_IQ2XS: tuple[int, ...]` — the 128-entry sign codebook used by
  IQ2_XS encoders (parity-bit convention from gguf-py).
- `KVALUES_IQ4NL: tuple[int, ...]` — the 16-entry IQ4_NL / IQ4_XS 4-bit
  signed-level table.

Required accessors (lazy, per-(device, dtype) cached):

- `get_iq2xs_grid(device, dtype) -> torch.Tensor` — expands
  `IQ2XS_GRID_RAW` to a `[N, 8]` magnitude table on the requested
  device/dtype the first time it is asked for, and caches the tensor.
- `get_ksigns_iq2xs(device, dtype) -> torch.Tensor` — same caching pattern.
- (`KVALUES_IQ4NL` is short enough to materialise on demand without
  caching; consumers convert it to a tensor at the call site.)

The module owns the codebook *values* and access caching; it owns no
encoding logic. Encoding STEs live in LLR-0054.

## Rationale

The IQ-family GGUF quants are codebook-based: an encoder cannot be
written without these tables, and inventing values silently produces
artifacts that do not round-trip through ggml's reference decoder. The
spec-review explicitly called out (C2) that IQ2_XS encoding requires
*both* a magnitude grid and a sign codebook — a magnitude-only argmin
flips negative values.

Centralising the constants in one module with a pinned-SHA docstring is a
deliberate audit posture: any future change to a codebook must update
the SHA, which surfaces in the diff. Per-(device, dtype) tensor caching
avoids reallocating constants on every STE invocation; the cache key is
intentionally narrow (one tensor per device/dtype combination, not per
shape) because the codebooks are constants.

## Acceptance Criteria

- [ ] Module docstring records the upstream commit SHA and the names of
      the C/Python files the constants were transcribed from
- [ ] `IQ2XS_GRID_RAW`, `KSIGNS_IQ2XS`, `KVALUES_IQ4NL` are
      module-level `tuple[int, ...]` constants (not lists; not mutable)
- [ ] `get_iq2xs_grid(device, dtype)` returns a `torch.Tensor` of shape
      `[512, 8]` (expanded magnitude grid) on the requested device/dtype
- [ ] `get_ksigns_iq2xs(device, dtype)` returns a `torch.Tensor` of
      shape `[128, 8]` on the requested device/dtype (each row expands
      one byte into 8 ±1 floats per ggml's bit-pattern convention)
- [ ] Repeated calls with the same `(device, dtype)` return the SAME
      tensor object (identity), not a fresh allocation
- [ ] A transcription spot-check test compares a handful of grid entries
      (and the parity bits of a handful of ksigns entries) to literals
      copied from the same upstream commit, so codebook drift fails the
      test rather than silently corrupting quants
